# 方向矫正文档：把主语从 Benchmark 调整为 Agent

## 1. 真正目标

**做一个能展示 AI Agent 能力的项目，用于应聘 AI Agent 相关岗位。**

更准确地说，我们要把当前项目收口成：

> **一个面向真实 GitHub issue 的 coding agent 项目，内置强验证底座。**

面试官在 5 分钟内，应该能回答三个问题：

1. 这个 agent 做了什么？
   - 修真实 GitHub issue
2. 它是怎么做的？
   - 读 issue -> 搜代码 -> 理解 bug -> 写 patch -> 跑测试验证 -> 失败时自我修正
3. 怎么证明它做得好？
   - 有规则版 baseline
   - 有真实任务验证集
   - 有 frozen 回归保护
   - 有 case study 和对比结果

**当前项目的核心问题不是“工程做得不够强”，而是“项目主语错位了”。**

现在仓库里最强的是：

- benchmark / eval / harness / tracking
- semi-real 真实 issue 任务池
- 规则版求解器

但如果目标是找 **agent 实习**，面试官首先想看到的应该是：

- agent 本体是什么
- agent 如何决策
- agent 如何调工具
- agent 如何失败恢复

所以真正要做的不是推翻现有项目，而是：

> **保留现有验证底座，把项目叙事和实现重点重新切回 Agent 本体。**

## 2. 当前项目状态：严格审计

这部分不做情绪判断，只做事实判断。

### 2.1 当前已经非常强的资产

| 资产 | 当前状态 | 在新叙事中的角色 |
|---|---|---|
| semi-real 真实 issue 任务集 | 正式集 `66` 条，challenge 集 `6` 条 | Agent 的验证集与边界案例池 |
| 候选池 | 当前 `72` 条，已形成状态流转 | 待修复 issue 队列 |
| 6 个核心工具 | `list_files / search_code / read_file / run_tests / write_file / show_diff` | Agent 的“眼睛”和“手” |
| Harness | workspace 隔离、run 目录、trace / result / patch 落盘 | Agent 的安全运行沙箱 |
| Pydantic schemas | `Task / Trace / Result` 已稳定 | Agent 输入输出与日志结构 |
| frozen sets | `frozen_20 / frozen_40` 已建立，`frozen_40_streak = 8` | Agent 版本回归保护 |
| 规则版 patcher / solver | 当前正式集可稳定通过 | **Rule-based baseline**，不是废资产 |
| 批量评测 / taxonomy / compare / stability | 已可直接使用 | Agent 质量门控层 |

### 2.2 当前真正的问题

当前不是“Agent 完全不存在”，而是：

> **当前存在的是 rule-based solver / baseline，不是你要拿去展示的 LLM coding agent。**

这句话很重要，因为它决定了接下来的路线不是从 0 开始，而是从：

- `已有规则版 baseline`
- `已有任务池`
- `已有 eval 与 harness`

往上补出：

- `LLM coding agent`

也就是说，当前项目缺的不是地基，而是**主角**。

### 2.3 当前叙事为什么不对齐求职目标

如果现在把项目直接拿去投递 AI Agent 岗位，面试官更容易看到的是：

- 你搭了一套 benchmark 基础设施
- 你做了很多任务管理、追踪、评测和稳定性控制
- 你很像在做 eval / infra / benchmark 工程

而不是第一时间看到：

- 你做了一个 agent
- agent 能处理真实问题
- agent 有清晰的决策闭环

所以当前项目的偏离，不是技术偏离，而是**叙事与展示重心偏离**。

## 3. 哪些资产应该保留，哪些应该降级

### 3.1 直接保留并升格

这些资产不是负担，反而是你区别于普通“做了个 demo agent”的护城河：

| 资产 | 处理方式 | 原因 |
|---|---|---|
| semi-real task / repo 池 | 保留 | 真实验证集很难临时补出来 |
| Harness | 保留 | 这是 Agent 落地运行的核心设施 |
| 工具层 | 保留 | LLM agent 可以直接复用 |
| frozen_20 / frozen_40 | 保留 | 这是回归保护，不该丢 |
| rule-based patcher | 保留并重命名为 baseline | “规则版 vs LLM 版”是很强的项目叙事 |
| batch eval / compare / stability | 保留但降级为验证层 | 证明 Agent 好不好，而不是项目主角 |

### 3.2 需要重新包装

| 当前叫法 | 更合适的叫法 |
|---|---|
| Benchmark 评测系统 | Agent 质量门控层 |
| 正式 benchmark 任务集 | Agent 验证集 |
| Challenge set | Agent 边界案例集 |
| Candidate pool | 待修复 issue 队列 |
| Maturity audit | Agent 版本发布检查 |

### 3.3 不应再继续作为主线投入的部分

这些不是“没用”，而是**不应该继续占据项目主线**：

| 资产 | 当前问题 | 处理方式 |
|---|---|---|
| `GUIDE.md` 超长 phase 日志 | 信息太多，不适合作为求职主文档 | 保留，但不再作为主展示面 |
| `optimization_log.md` 超长版本史 | 内部记录价值大，对外展示价值低 | 保留，降级为归档型材料 |
| maturity / tracking 大量辅助脚本 | 已经足够成熟 | 保留可用，不继续作为主线讲述 |
| benchmark 继续无上限扩容 | 边际收益快速下降 | 从“规模驱动”改成“案例驱动” |

## 4. 叙事翻转：从“我做了 benchmark”到“我做了 agent”

### 4.1 项目定位（改前 vs 改后）

**改前：**

> 一个面向 GitHub issue 自动修复的 mini agentic software engineering benchmark 项目

**改后：**

> 一个面向真实 GitHub issue 的 coding agent。给定 issue 描述，自动完成：理解问题 -> 检索代码 -> 定位根因 -> 生成补丁 -> 测试验证 -> 失败后再次修正。内置验证层覆盖 `66` 条正式真实任务、`6` 条 challenge 边界任务、规则版 baseline 对比，以及 frozen 回归保护。

### 4.2 指标展示（改前 vs 改后）

**改前：**

```text
| 正式真实任务数 | 66 |
| challenge 任务数 | 6 |
| 来源生态数 | 16 |
| frozen_40 连续无回归版本数 | 8 |
```

**改后：**

```text
| Agent 能力 | 指标 |
| 端到端修复成功率 (LLM agent) | X% |
| vs 规则版 baseline | X% vs baseline |
| 平均修复步数 | X |
| 失败后二次修复率 | X% |
| 回归保护 | frozen_40 连续不退化 |
```

### 4.3 项目故事线（改前 vs 改后）

**改前故事线：**

> 我做了一套 benchmark 体系，有冻结集、稳定性复跑、成熟度审计，能评测 agent。

**改后故事线：**

> 我做了一个面向真实 GitHub issue 的 coding agent。它会读 issue、搜代码、改补丁、跑测试、必要时再修一次。为了验证它不是 demo，我又构建了一套真实任务验证层、规则版 baseline 和 frozen 回归保护。

**关键区别：**

- Agent 是主角
- benchmark 是证据
- harness 是底座

而不是反过来。

## 5. 方向矫正后的核心判断

### 5.1 当前最重要的事实

**当前项目已经有了“验证层”，但还没有“可展示的 LLM Agent 主体”。**

因此接下来最重要的工作不是：

- 再多扩 10 条 benchmark
- 再做一轮 maturity audit
- 再做一套新的 tracking

而是：

> **把真正的 LLM coding agent 补出来，并让它跑在现有底座上。**

### 5.2 当前不应该做的误判

以下判断都不够准确：

- “现在项目没价值了”
- “前面做 benchmark 都白做了”
- “必须推倒重来”

更准确的说法是：

> **前面做的东西多数仍然很值钱，但需要被重新放到正确的位置：它们是 Agent 的底座和证据，而不是项目主角。**

---

## 6. 接下来该做什么：按优先级

## P0：让真正的 LLM Agent 存在

这是唯一不可跳过的步骤。

当前仓库里主求解器仍然是规则版 baseline。下一步必须补出一个真正的 LLM Agent。

要做的事：

1. 接入 LLM SDK 与配置层
   - 可以是 deepseek/kimi / Anthropic SDK 等
2. 把现有 6 个工具包装成 LLM 可调用工具
3. 实现最小 ReAct 循环
   - LLM 决策 -> 调工具 -> 读取结果 -> 再决策
4. 复用现有 harness
   - workspace 隔离
   - trace / result 落盘
5. 在 3 到 5 个简单任务上先跑通闭环

目标产物：

- `app/agent/llm_agent.py`
- `app/agent/llm_prompts.py`
- `app/agent/tool_definitions.py`
- `scripts/run_issue_agent.py`

## P1：明确“双轨结构”

要把项目从结构上讲清楚：

- `rule_based_agent` / `patcher.py`
  - 规则版 baseline
- `llm_agent`
  - 真正要展示的主角
- `benchmarks / evals / frozen`
  - 验证与回归保护层

这一步需要改：

1. README 第一屏
2. 架构文档
3. 命名与入口

## P2：让 Agent 可展示、可理解

这一层是求职很关键的部分。

需要新增：

1. `docs/agent_overview.md`
   - 架构、工作流、关键决策
2. `docs/agent_case_studies.md`
   - 3 个成功 / 失败案例
3. `scripts/run_issue_agent.py`
   - 一个简洁的使用入口
4. 更可读的 trace / summary
   - 让“Agent 是怎么思考和修复的”能被看懂

## P3：让 Agent 更像 Agent

等最小闭环跑通后，再补：

1. Reflection
   - post-test 失败后读错误，再修一次
2. Prompt 迭代
   - 用失败案例改 prompt
3. 小规模模型对比
   - 不一定要马上做大规模
4. 可选 verifier / reviewer 角色
   - 不是现在的 P0，但后面可以考虑

## P4：验证层继续服务主线

验证层不停止，但要降级为“服务 Agent 主线”：

1. 跑一版 frozen_20
2. 跑一版正式集小样本
3. 输出 LLM agent vs baseline 的对比
4. 保留 frozen_40 作为后续版本回归门

## 7. 哪些事情现在不该继续当主线

| 不该继续当主线的事 | 原因 |
|---|---|
| 继续以 `v73 / v74` 为目标扩 benchmark | 当前最缺的是 Agent 主体，不是更多数据点 |
| 持续维护超长优化日志作为核心产出 | 对求职展示帮助有限 |
| 把 challenge 扩容当作最优先任务 | challenge 目前已有 `6` 条，够展示边界意识 |
| 过早做 multi-agent | 单 agent 闭环还没成为主角，过早做会分散精力 |
| 过早做复杂 UI | 当前 CLI + markdown trace 已够展示 |
| 为了“像 agent”而强行引入重框架 | 先做透明、可讲清的最小闭环更重要 |

需要注意：

这不等于“以后永远不做”，而是：

> **在 LLM Agent 主线完成之前，这些都不应抢占最高优先级。**

## 10. 一句话结论

> **从现在开始，这个项目的主语应该是 Agent。**
>  
> Benchmark、Harness、Frozen Set、Tracking 都继续保留，但它们的角色应明确降为：
> **服务 Agent、验证 Agent、保护 Agent，而不是替代 Agent 成为项目主角。**
