# Harness 设计说明

本项目会明确把工程重点放在 `Harness`，而不是把大量规则硬编码成“伪智能流程图”。

这个思路参考了本机上的 `learn-claude-code` 项目，尤其是：

- 工具分发
- 路径安全
- 任务持久化
- 工作区隔离
- 错误恢复

## 什么是我们这里的 Harness

在这个项目里，Harness 主要由五部分组成：

- `Tools`：文件、搜索、测试、patch、diff
- `Observation`：代码内容、测试结果、diff、trace
- `Persistence`：`task.json`、`result.json`、`trace.json` 等落盘产物
- `Isolation`：每次 run 的独立工作副本
- `Permissions`：路径边界、写入范围、后续可能的命令限制

模型负责：

- 理解 issue
- 形成修复假设
- 决定下一步读什么、测什么、改什么

Harness 负责：

- 提供工具
- 约束边界
- 保存状态
- 记录证据
- 保障可评测和可复现

## 从 learn-claude-code 借鉴的核心优点

### 1. Agent Loop 保持简洁

借鉴点：

- 不把主循环写成复杂的硬编码决策树
- 把复杂度放在工具、状态和恢复机制上

在本项目中的落地：

- `app/agent/*` 只负责最小决策与执行组织
- `app/tools/*` 负责能力边界
- `app/runtime/*` 负责状态与运行时管理

### 2. 工具分发采用显式接口

借鉴点：

- 每个工具有明确名称、输入、输出、错误类型
- 工具层和模型层解耦

在本项目中的落地：

- 所有工具统一返回 `ok / tool_name / summary / data / error`
- 后续可在此基础上追加 tool registry，而不改 Agent loop 结构

### 3. 路径安全前置

借鉴点：

- 先做 `safe_path` 约束，再做文件工具

在本项目中的落地：

- 所有文件读写最终都要限制在当前 run workspace 内
- benchmark 原仓库只读保留
- 后续 `write_file` 和 patch 应用会复用同一套边界检查

### 4. 状态文件是真实状态源

借鉴点：

- 任务状态和运行状态都应持久化到磁盘
- 审计事件可以附加，但不替代主状态文件

在本项目中的落地：

- 单次 run 的主状态文件固定为：
  - `task.json`
  - `result.json`
  - `trace.json`
  - `patch.diff`
  - `summary.md`
- 未来就算新增事件日志，也只作为补充审计信息

### 5. 隔离优先于共享工作区

借鉴点：

- 不同任务不应共享同一可写目录
- 目录隔离会直接降低状态污染和误改风险

在本项目中的落地：

- 每条任务都要复制出独立 `workspace/`
- 所有修改只允许发生在该 `workspace/`
- 后续如果需要并行运行，再考虑更强的 worktree 策略

### 6. 恢复机制是 Harness 的一层，而不是循环里到处打补丁

借鉴点：

- 错误恢复应与任务系统、工具系统解耦

在本项目中的落地：

- 先完成主线最小闭环
- 后续再单独增加：
  - API 重试
  - 上下文压缩
  - 测试失败后的有限重试
  - 失败退出策略

## 当前阶段的落地范围

当前只在 `Phase 0` 和即将开始的 `Phase 1` 中落地最必要的部分：

- 目录骨架固定
- schema 固定
- run 产物路径固定
- 路径边界工具准备

当前已经进一步落地：

- 观察型工具
- 测试执行工具
- 测试 stdout / stderr 落盘
- 测试失败摘要提取
- patch 写入工具
- 原仓库与 workspace 的统一 diff
- 单任务 patch 闭环

以下能力暂不提前实现：

- 复杂任务图
- worktree 并行隔离
- 多 agent 协作
- 复杂恢复状态机
- OpenTelemetry 全量埋点

原因很简单：

- 当前主线还处于早期
- 先把单任务最小闭环做通，比提前堆复杂机制更重要

## 当前技术建议

### Pydantic

建议：`可以引入，而且比较值得`

适合引入的位置：

- `Task / Trace / Result / Config` 结构校验
- 运行配置序列化
- 后续评测结果结构化输出

当前建议：

- 已在 `Phase 1` 接入 `Task / Trace / Result` schema
- 优先只用于“边界数据”，不要把业务逻辑写进 model 层

### OpenTelemetry

建议：`现在不需要立刻引入`

原因：

- 当前还没有稳定的 Agent run 主链路
- 过早接入会增加实现复杂度和维护负担
- 现在先把本地 JSON trace 做扎实，收益更高

更合适的引入时机：

- `Phase 4` 之后，批量运行与评测稳定时
- 或者当你明确需要接 Prometheus / Jaeger / Langfuse / APM 时

## 后续落地方向

接下来会沿着这个顺序推进：

1. 先做基础工具
2. 再做测试闭环
3. 再做 patch 闭环
4. 再做 run 目录与批量结果组织
5. 最后再考虑更强的任务系统、恢复系统和观测系统
