# GitHub 真实 Issue 筛选规范

本文件用于给外部检索器或协作模型一个稳定、可执行的 issue 搜索标准。

目标不是“找到有趣的 issue”，而是“找到适合转成 benchmark 的 issue”。

## 核心目标

我们要找的是这类 issue：

- 能被缩减成一个最小可复现 bug
- 能被改写成一个小型 `semi_real` benchmark
- 修复边界清晰，适合做 `bugfix` 评测
- 最好能在 1 个函数或 1 个模块内完成修复

## 必须满足的要求

### 1. 仓库要求

- 优先 Python 仓库
- 优先中小型仓库
- 依赖不要太重
- 测试环境最好只需 `pytest` 或非常少量依赖
- 不要优先选择需要数据库、浏览器、服务编排、系统权限或外部 API 的仓库

### 2. Issue 类型要求

- 必须是 `bug` 或明确的行为回归
- 必须有清晰的“当前错误行为”和“期望正确行为”
- 最好自带最小复现代码、输入输出样例、失败现象或报错信息
- 最好能归结为：
  - 单函数逻辑错误
  - 边界条件错误
  - 解析错误
  - 格式化错误
  - 继承/优先级/控制流判断错误
  - 容器/集合/切片/序列化行为错误

### 3. 修复边界要求

- 最好只改 1 个文件
- 可以接受改 2 个文件，但核心逻辑文件必须非常明确
- 不要选择需要大范围重构的 issue
- 不要选择需要改公共 API 设计、协议设计、架构层抽象的 issue

### 4. 测试要求

- 最好能直接写成 1 到 3 个回归测试
- 失败测试要稳定、确定
- 不要依赖时间竞争、网络波动、随机性、平台特性
- 不要是“偶发失败”或“需要多轮重试才会出现”的问题

### 5. benchmark 友好性要求

- 适合还原成最小 repo
- 适合写成清晰的 `issue_title`
- 适合生成 `success_criteria`
- 适合映射到现有 patch 策略能力扩展

## 明确排除的 issue

以下 issue 不要作为优先候选：

- 纯文档问题
- 需求讨论
- API 设计争议
- 性能优化但没有清晰正确性 bug 的问题
- 需要跨多个子系统联动的大修
- 需要外部服务、云资源、数据库、浏览器环境才能稳定复现的问题
- 平台相关性极强的问题
- 语义含糊，社区讨论里都无法确认“到底是不是 bug”的问题
- 更像“规范解释”而不是“实现错误”的问题

## 理想 issue 模板

最理想的 issue 往往长这样：

- 标题直接描述错误行为
- 有一段最小复现代码
- 有明确输入
- 有明确当前输出
- 有明确期望输出
- 能定位到一个较小的目标模块
- 修复后很容易补回归测试

## 优先级排序规则

当 Claude 找到多个候选时，请按下面优先级排序：

1. 边界最清晰、最容易最小化复现的 bug
2. 单函数或单模块即可修复的 bug
3. 测试最容易写、最稳定的 bug
4. 与现有 benchmark 类型不同、能增加语义多样性的 bug
5. 仓库体量更小、依赖更轻的 bug

## 返回格式要求

让 Claude 返回候选列表时，请严格按下面结构整理：

```text
1. repo: owner/name
   issue: #1234
   title: ...
   url: ...
   why_it_fits:
   expected_target_files:
   expected_test_shape:
   estimated_difficulty: easy / medium / hard
   risk_notes:
   recommendation: high / medium / low
```

## 当前最匹配的方向

基于我们当前 benchmark 的演进，后续优先欢迎这几类 issue：

- 解析类 bug
- 格式化类 bug
- 切片 / 容器 / 序列化类 bug
- 优先级 / 继承 / 覆盖顺序类 bug
- 控制流分支判断类 bug
- 小型模板、配置、字符串处理类 bug

## 当前不优先的方向

- 需要大量领域知识的复杂框架 bug
- 需要真实仓库大环境才能复现的问题
- 只靠 issue 描述无法判断正确行为的问题
- 修复需要改一大串文件的问题
