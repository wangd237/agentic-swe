## 工具路由优化 v1：Phase-based Schema Filtering

### 优化思路

当前 baseline 的工具路由分为两层：

- **Schema 层**：`app/agent/tool_definitions.py` 中统一定义 10 个工具 Schema；`app/agent/llm_agent.py` 在每轮 LLM 调用时都传入 `build_tool_definitions()` 返回的全量工具列表。因此模型在 `understand`、`reproduce`、`localize`、`patch`、`verify` 等阶段都会看到全部 10 个工具。
- **执行层**：`app/agent/tool_policy.py` 通过 `ALLOWED_TOOLS_BY_PHASE` 和 `ToolPolicy.validate()` 按当前 phase 拦截非法工具调用。例如模型即使在非 patch 阶段请求 `write_file` / `edit_file`，执行层也会阻止。

baseline 的问题是：执行层能保证安全，但无法减少传给模型的工具 Schema token，也无法避免模型先看到无关工具后再产生不合阶段的工具调用。

本次优化将一部分路由能力前移到 Schema 层：

- **Schema 层**：新增 `app/agent/tool_router.py`，每轮 LLM 调用前根据当前 `agent_state.phase` 调用 `build_tools_for_phase()`，只向模型暴露当前 phase 在 `ALLOWED_TOOLS_BY_PHASE` 中允许的工具 Schema。
- **执行层**：继续保留 `ToolPolicy.validate()` 作为安全兜底。即使模型因为历史上下文等原因请求了当前 phase 不允许的工具，执行层仍然会拦截。

优化后的路由概况：

`Schema 层：按当前 phase 仅传 ALLOWED_TOOLS_BY_PHASE 允许的工具 Schema；执行层：继续由 ToolPolicy.validate() 按 phase 和 patch gate 拦截非法工具调用`

## 工具路由优化 v2：Phase + State-aware Filtering

### 优化思路

v1 已经将工具路由从“每轮全量传 10 个工具 Schema”改为“按当前 phase 只传允许工具 Schema”。v2 在此基础上继续引入确定性的 state-aware 过滤：不只看 `agent_state.phase`，还看当前 workspace 和验证状态，进一步隐藏当前状态下明显无效或会被执行层拦截的工具。

本次新增两条规则：

- 当 `workspace_generation == 0` 时，不向模型暴露 `undo`。此时 workspace 尚无修改，模型没有可回滚内容。
- 当处于 `verify` 阶段，且当前 patch 尚未通过 `show_diff` 观察时，不向模型暴露 `run_tests`。这与执行层 `ToolPolicy.requires_diff_before_tests()` 保持一致，将“必须先 show_diff 再 run_tests”的约束前移到 Schema 层。

执行层仍然保留 `ToolPolicy.validate()` 作为安全兜底。v2 的目标不是改变 agent 行为边界，而是在不牺牲安全性的前提下，减少模型每轮可见工具数量和工具 Schema token。

优化后的路由概况：

`Schema 层：按当前 phase 和 AgentState 仅传当前可用工具 Schema；执行层：继续由 ToolPolicy.validate() 按 phase、patch gate 和 diff-before-tests 规则拦截非法工具调用`

## LLM 调用次数偏高的原因概括

当前 agent 仍以“每轮询问 LLM 下一步动作”为主循环，导致部分确定性流程也需要模型参与决策。典型表现是：复现失败后由模型决定读取上下文，修改后由模型决定查看 diff，查看 diff 后由模型决定运行测试，测试通过后再由模型决定总结或结束。也就是说，系统已经具备 phase、policy 和验证状态，但仍有不少固定工程流程停留在 LLM 决策层，造成调用次数偏高。

## 工具路由优化 v3：Immediate Post-write Auto Verification

### 优化思路

v3 将补丁写入后的确定性验证流程前移到 runtime：当 `write_file` / `edit_file` 成功后，系统立即自动执行 `show_diff` 和 `run_tests`，并将 diff 与测试结果和写入工具结果一起回传给模型。这样模型不再需要额外一轮来决定“先看 diff”，也不需要再一轮决定“运行测试”。

本次优化保留原有最终兜底验证逻辑：如果立即自动验证没有触发，系统仍会在任务结束前检查未验证改动并执行验证。

优化后的路由概况：

`执行流：write_file/edit_file 成功后 runtime 立即执行 show_diff + run_tests；Schema 层：继续按 phase 和 AgentState 过滤工具；执行层：继续由 ToolPolicy.validate() 兜底`

## 工具路由优化 v4：Failure Context Auto-load

### 优化思路

v3 已经把补丁写入后的 `show_diff + run_tests` 变成 runtime 自动执行，明显减少了 LLM 在验证阶段的决策轮次。v4 继续沿着“把确定性工程动作从 LLM 决策层下沉到 runtime”的方向推进：当 `run_tests` 返回失败，并且失败摘要中包含明确的文件位置时，runtime 自动读取失败位置附近代码，并把这段上下文作为下一轮模型输入的一部分。

本次优化的目标是减少模型在测试失败后反复询问“应该读哪个文件 / 哪几行”的决策成本，提高失败诊断阶段的 grounding。

优化后的路由概况：

`执行流：run_tests 失败后 runtime 根据 failure_summary.locations 自动 read_file 失败位置上下文；Schema 层：继续按 phase 和 AgentState 过滤工具；执行层：继续由 ToolPolicy.validate() 兜底；验证流：保留 v3 的 write_file/edit_file 后立即 show_diff + run_tests`

## 工具路由优化 v4.1：Conditional Failure Context Auto-load

### 优化思路

v4 的问题是触发过于积极：只要 `run_tests` 失败且输出中存在失败位置，就自动读取失败位置上下文。对于简单任务，如果模型已经读取过测试文件，且失败位置只是测试断言行，再次自动读取同一个测试文件会增加 token，但不一定减少 LLM 调用次数。

v4.1 将自动失败上下文读取改为条件触发：

- 如果失败发生在补丁之后，仍自动读取上下文，辅助修正失败补丁。
- 如果失败位置包含多个文件，自动读取上下文，辅助多点定位。
- 如果失败位置指向非测试文件，自动读取上下文，优先补充实现侧 grounding。
- 如果失败位置只指向测试文件，且该测试文件已经被模型读取过，则跳过自动读取，避免重复上下文。

优化后的路由概况：

`执行流：run_tests 失败后 runtime 先判断 failure_summary.locations 与 AgentState.read_files，仅在高收益场景自动 read_file；Schema 层：继续按 phase 和 AgentState 过滤工具；验证流：继续保留 v3 的 write_file/edit_file 后立即 show_diff + run_tests`

## 当前代码基线声明：回退到 v3

v4 / v4.1 的实验记录保留，用于说明失败上下文自动加载方向的验证过程和量化结果；但当前代码实现已回退到 v3，即保留 `phase + state-aware` 工具 Schema 过滤和 `write_file/edit_file` 后立即自动 `show_diff + run_tests` 的机制，不再默认启用失败上下文自动加载。

回退原因：

- v4 / v4.1 没有证明能稳定降低 token 消耗。全 6 任务对比中，v4 相比 v3 增加 `346 tokens`，v4.1 相比 v3 增加 `2,738 tokens`。
- v4 在 `task_019` 上曾减少 1 次 LLM 调用，但 v4.1 复跑未复现，说明该收益可能来自单次 LLM 输出波动，而不是稳定机制收益。
- 失败上下文自动加载会额外注入代码上下文；当失败位置只是测试断言、或模型已经读过相关文件时，这部分上下文容易变成额外 token 成本。
- 当前主目标是 token 友好和调用次数可控，v3 的收益更稳定：它通过减少确定性验证步骤中的 LLM 决策轮次，在 5 个扩展任务上相比 baseline 降低约 `27.4%` token。

因此，当前主线继续以 v3 为代码基线；v4 / v4.1 暂不进入默认实现，后续如需继续研究，应作为显式实验开关进行 A/B 对照，而不是默认策略。

## 工具路由优化 v5：Verification Dedup + Tool Result Compression

### 优化思路

v5 不再沿用 v4 / v4.1 的失败上下文自动加载方向，而是回到更确定的 token 友好目标：减少重复验证和压缩工具结果。

本次实现包含两部分：

- **Verification Dedup**：当当前 workspace generation 已经通过 full verification 后，如果模型再次请求等价的 full `run_tests`，runtime 直接返回一个短的 dedup 结果，不再重复启动 pytest。
- **Tool Result Compression**：`show_diff` 回传给模型时保留 `changed_files`、`diff_char_count` 和截断后的 `diff_text`，长 diff 不再默认完整塞回模型上下文，并提示后续如需精确上下文应使用 `read_file`。

优化后的路由概况：

`执行层：已 full verify 通过的同一 workspace generation 跳过重复 run_tests；工具结果层：压缩 show_diff 长 diff；Schema 层：继续使用 v3 的 phase + AgentState 过滤`

## 下一阶段方向 v6：Patch Preflight / Edit Reliability

### 优化思路概述

v5 的实验暴露出一个更关键的问题：token 消耗大幅上升往往不是来自工具 Schema 或单次工具结果过长，而是来自错误编辑导致的返工。例如 `task_022` 中模型先产生了一次失败的 `edit_file`，随后又重新读取文件、重新生成补丁，最终造成额外 LLM 轮次和 token 消耗。

因此，v6 的优化重点应从“继续压缩上下文”转向“减少错误补丁动作”，尤其是降低 `edit_file` 的 `old_string_not_found` 这类失败。

这里的 `old_string` 指 `edit_file` 工具要求模型提供的“原文件中要被替换的精确原文”。例如模型希望把 `return 1` 改成 `return 2`，就需要传入目标文件里完全一致的 `old_string` 和替换后的 `new_string`。如果缩进、空格、换行或上下文不一致，工具就无法安全替换，并返回 `old_string_not_found`。这类失败说明模型的修改意图可能是对的，但引用的原文不够精确，容易导致额外的读取文件、重新生成补丁和再次验证。

初步方向：

- 当 `edit_file` 因 `old_string_not_found` 失败时，工具结果中附带目标文件的相似上下文片段，帮助模型下一轮直接修正 `old_string`。
- 在执行 `edit_file` 前增加轻量 preflight：如果 `old_string` 明显不在目标文件中，优先返回可操作的定位反馈，而不是只给一个失败结果。
- 对 patch 阶段的编辑动作增加更强的“基于当前文件片段编辑”约束，鼓励模型先读取最新上下文，再生成精确 patch。

预期收益：

v6 不直接以压缩单轮 token 为目标，而是减少失败工具调用和返工轮次。相比 v5，这条线更贴近当前 token 波动的主要来源：一次错误编辑可能带来一整轮甚至多轮额外 LLM 调用，其成本远高于压缩几百 token 的工具结果。

### v6 最小实现与首次验证

本次先实现低风险版本：不改变成功编辑路径，只增强失败编辑路径。

- `edit_file` 在 `old_string_not_found` 时返回 `similar_contexts`，包含相似度、行号范围、`suggested_old_string` 和上下文片段。
- `ToolExecutor.summarize_for_model()` 对失败的 `edit_file` 保留 `similar_contexts` / `matches`，去掉无关字段，保证模型能直接看到可修正的候选原文。
- 保留 v5 的 `show_diff` 压缩测试，但 v6 的主要目标转为 edit reliability。

首次验证选择 `task_022`，因为它在 v5 中出现过失败编辑返工：

| 版本 | Run ID | LLM 调用次数 | Token 消耗 | 工具调用数 | 失败 edit_file 次数 | 结果 |
| --- | --- | ---: | ---: | ---: | ---: | --- |
| v3 | `run_20260624T142204224604Z_4399` | 6 | 21,707 | - | - | success |
| v5 | `run_20260625T034903836860Z_6612` | 8 | 33,279 | 11 | 1 | success |
| v6 | `run_20260625T074136612324Z_9660` | 6 | 21,667 | 9 | 0 | success |

初步观察：

- v6 在 `task_022` 上回到 v3 水平：LLM 调用次数从 v5 的 `8` 降至 `6`，token 从 `33,279` 降至 `21,667`。
- 本次 v6 run 没有触发失败 `edit_file`，因此不能单独证明 `similar_contexts` 机制已经带来收益；但至少说明该实现没有伤害 `task_022`，且当前方向与 v5 暴露的问题更贴近。
- 下一步应挑选或构造包含 `old_string_not_found` 的样本，专门评估 `similar_contexts` 是否能减少返工轮次。

### v6.1 / v7：Edit Failure Recovery 验证

针对历史 run 中出现的 `old_string_not_found`，选取 `task_030` 和 `task_036` 做专门验证。

v6.1 的主要调整：

- **补丁恢复态**：当已经产生过 workspace 改动、已有复现证据和定位候选，并且反思回到 `localize` 时，Schema 层重新暴露 `edit_file / write_file / show_diff / undo`，避免模型想继续修补但被 phase policy 反复拦截。
- **编辑失败提示**：当 `edit_file` 返回 `old_string_not_found` 时，工具结果后追加 `EDIT_RECOVERY_NOTICE`，提醒模型优先使用 `similar_contexts[].suggested_old_string`，而不是继续凭记忆生成 old_string。
- **建议字段强化**：在保留 `similar_contexts` 的基础上，`ToolExecutor.summarize_for_model()` 增加 `recommended_next_old_string`，把最高相似候选提升为一等字段，降低模型忽略嵌套建议的概率。

验证结果：

| 任务 ID | 版本 | Run ID | LLM 调用次数 | Token 消耗 | 工具调用数 | 失败 edit_file | Policy blocked edit | 结果 |
| --- | --- | --- | ---: | ---: | ---: | ---: | ---: | --- |
| `task_030` | v6 | `run_20260625T075920437390Z_9778` | 12 | 78,653 | 19 | 1 | 3 | success |
| `task_030` | v6.1 | `run_20260625T081310584533Z_9218` | 9 | 51,489 | 16 | 1 | 0 | success |
| `task_036` | v6.1 | `run_20260625T081401177099Z_2925` | 8 | 37,762 | 11 | 2 | 0 | success |
| `task_036` | v7 safer recommendation | `run_20260625T082904461970Z_5657` | 8 | 36,869 | 11 | 1 | 0 | success |

定性结论：

- `task_030` 证明 v6.1 的补丁恢复态有效：原本在 failed verification 后，模型回到 `localize` 却连续尝试 `edit_file`，导致 3 次 policy blocked；v6.1 后不再发生该类拦截，LLM 调用从 `12` 降至 `9`，token 从 `78,653` 降至 `51,489`。
- `task_036` 证明单纯给 `similar_contexts` 还不够：v6.1 中模型连续两次产生 `old_string_not_found`。加入 `recommended_next_old_string` 后，失败编辑从 `2` 次降为 `1` 次，token 小幅下降。
- 期间尝试过两条失败路线，均不进入默认实现：
  - **重复失败拦截**：拦截同一文件同一 `old_string` 的重复失败编辑，但会迫使模型多思考一轮，`task_036` 从 `8 calls / 37,762 tokens` 变为 `9 calls / 43,785 tokens`。
  - **高相似自动修正写入**：工具层直接用高相似建议替换 old_string，虽然减少了 `old_string_not_found`，但可能把错误补丁成功写入，`task_036` 变为 `9 calls / 55,694 tokens / 17 tool calls`，风险高于收益。

当前保留方案：

`Schema 层：phase + AgentState 过滤；补丁恢复态下允许 localize 继续暴露写工具；执行层：不自动代替模型写入，仅在 old_string_not_found 时返回 similar_contexts 和 recommended_next_old_string；验证层：继续保留写入后立即 show_diff + targeted/full run_tests`

## 工具路由优化 v8：Reproduce Read-only Search

### 优化思路

`pydicom__pydicom-1139` 暴露出一个新的路由问题：`reproduce` 阶段虽然已经有失败复现证据，但 Schema 层和执行层都只允许 `read_file / run_tests / show_diff`，不允许 `grep/search_code`。当模型需要从错误描述中的类名、旧类名或方法名定位实现位置时，只能反复读取文件片段，导致迭代浪费。

v8 将 `grep/search_code` 加入 `reproduce` 阶段允许工具：

- **Schema 层**：`reproduce` 阶段向模型暴露 `grep/search_code/read_file/run_tests/show_diff`。
- **执行层**：`ToolPolicy.validate()` 同步允许 `grep/search_code`，但仍不允许 `edit_file/write_file`。
- **安全边界**：本次只放开只读搜索工具，不改变 patch gate，不允许 reproduce 阶段写文件。

优化后的路由概况：

`Schema 层：reproduce 阶段允许只读搜索 grep/search_code；执行层：同步允许只读搜索但继续阻止写工具；patch gate 和 diff-before-tests 规则不变`

## 工具路由优化 v9：Failure Summary / Reproduction Signal Extraction

### 优化思路

v8 解决了 `reproduce` 阶段“能不能搜索”的问题，但 `pydicom__pydicom-1139` 还暴露出另一个信息质量问题：本地最小复现命令失败时，`run_tests` 原本只能给出泛化摘要：

`测试命令执行失败，但未能从输出中提取明确失败位置。`

实际上输出中已经包含关键异常信号：

`TypeError: argument of type 'PersonName' is not iterable`

v9 的目标是增强失败摘要解析，让非 pytest traceback / `python -c` 复现命令也能被结构化提取，降低模型从长输出里自己找信号的成本。

本次新增字段：

| 字段 | 含义 |
| --- | --- |
| `exception.type` | 从 traceback 最后一行提取异常类型，如 `TypeError` |
| `exception.message` | 异常消息，如 `argument of type 'PersonName' is not iterable` |
| `possible_symbols` | 从异常消息中提取可能有定位价值的符号，如 `PersonName` |

工具结果回传调整：

- `run_tests._build_failure_summary()` 在失败时提取 `exception` 和 `possible_symbols`。
- `ToolExecutor._compact_failure_summary_for_model()` 保留这两个字段，确保模型能直接看到。
- `FailureSignature.from_failure_summary()` 将这两个字段纳入失败签名，避免不同异常被误判为同一失败。

## 工具路由优化 v10：Failure Signal Guided Search

### 优化思路

v9 已经能从非 pytest traceback 中提取 `possible_symbols`，但这些符号只是作为上下文交给模型。v10 尝试将其变成 runtime 的确定性只读动作：

- 当 `run_tests` 失败；
- 且 `failure_summary.locations` 为空；
- 且存在 `possible_symbols`；
- runtime 自动对符号执行一次 `search_code`；
- 将搜索结果写回 `failure_summary.guided_search`，并把命中文件加入 localization candidates。

安全边界：

- 只执行只读 `search_code`。
- 不自动读取文件。
- 不自动修改文件。

## Agent Core Refocus v11：Verification Quality Layer

### 为什么下一步是这个方向

前面 v8-v10 主要优化了工具路由、失败信号抽取和自动只读搜索。这些能力都属于 agent core，但它们也暴露出一个更关键的问题：本地 smoke 成功不等于真实修复成功。

当前项目如果只证明“agent 能改代码并让一个最小复现命令通过”，与现有 Code Agent 的差异并不够强。真正的差异应该是：

`不是更会写代码，而是更可控、更可验证、更可审计。`

因此下一步应该优先补 **Verification Quality Layer**，原因如下：

- **真实业务需要可信结果**：用户关心的不是 agent 是否生成 patch，而是 patch 是否真的修复问题、是否有回归风险。
- **与普通 Code Agent 拉开差异**：成熟 Code Agent 已经能读写代码，本项目的差异应体现在阶段约束、验证强度、审计 trace 和风险说明。
- **避免 benchmark 方向跑偏**：SWE-bench Lite smoke 是测量工具，不是最终目标；必须明确区分 local smoke success、targeted success、full verification success 和 official resolved。
- **为真正多 Agent 铺路**：当前系统本质仍是单 repair agent + deterministic runtime。Verifier 是最适合作为第二个 agent/角色的边界，因为它天然负责审查 Repair Agent 的产物，而不是重复修代码。
- **防止 agent 自证成功**：Repair Agent 不应单独宣布“修复成功”；最终结果需要 verifier 层基于 patch、测试、验证强度和 caveat 给出结论。

### v11 最小实现方向

先实现确定性 Verifier Layer，而不是立即引入新的 LLM Verifier Agent。

输入：

- `final_status`
- `verification_strength`
- `patch_applied`
- `modified_files`
- `pre_test_exit_code`
- `post_test_exit_code`
- `source_type`
- task metadata

输出：

- `verification_level`
- `risk_level`
- `accepted`
- `caveats`
- `recommendations`

预期效果：

- result / summary 中清楚展示本次结果属于哪种成功等级。
- SWE-bench Lite smoke 自动带上 caveat：本地 smoke 通过不等于 official resolved。
- weak/static verification 不再只靠 `final_status` 命名区分，而是有结构化 verifier report。

### v11.1：accepted_final_status

v11 引入 `verifier_report` 后，仍然需要区分两个层面的状态：

- `final_status`：保留为 agent 原始执行状态，用于兼容既有评测脚本和历史日志。
- `accepted_final_status`：新增为产品层验收状态，用于告诉用户“这次结果是否真的可接受，以及属于哪种可接受/不可接受等级”。

这样做的原因是：`success` 在不同场景下含义并不相同。普通本地全量测试通过、SWE-bench Lite 本地 smoke 通过、只跑 targeted test 通过、以及 weak/static verification 通过，都不应该被用户理解成同一个“已可信修复”。

当前映射规则：

| verifier level | verifier accepted | accepted_final_status | 含义 |
| --- | --- | --- | --- |
| `official_resolved` | true | `accepted_success` | 官方 benchmark/harness 已确认 resolved |
| `full_verification_success` | true | `accepted_success` | 非 benchmark 场景下 full verification 通过 |
| `local_smoke_success` | false | `local_smoke_success` | SWE-bench Lite 本地 smoke 通过，但不能等同 official resolved |
| `targeted_success` | false | `targeted_only_success` | 只确认 targeted verification 通过，仍需 full verification |
| `weak_verification_success` | false | `weak_verification_success` | 只有弱验证/静态验证，不能产品层接受 |
| `none` | false | `not_accepted` | 未达到可信验证状态 |

预期效果：

- CLI、`result.json`、`summary.md` 和 trace final step 都能直接展示产品层状态。
- 后续 benchmark 对比仍可继续使用 `final_status`，产品展示和人工验收则优先看 `accepted_final_status`。
- 避免把 “agent 写了 patch 并让某个命令通过” 误报为 “真实业务可接受的修复”。

### v11.2：Verification Evidence Contract

v11 / v11.1 已经能输出验证结论，但还缺少一个结构化的“证据合同”。如果只有 `verifier_report`，用户能看到结论；但如果没有 `verification_evidence`，后续产品层、评测脚本或 Verifier Agent 还需要重新翻 trace 才能知道这个结论依据是什么。

v11.2 的目标是：不改变 agent 行为、不增加 LLM 调用、不增加工具调用，只把现有运行证据整理成机器可读结构。

新增字段：`verification_evidence`

核心内容：

- `patch_applied`：是否实际产生 patch。
- `modified_files`：本次 patch 影响的文件。
- `verification_scope`：本次验证强度，例如 `weak` / `targeted` / `full`。
- `pre_test`：patch 前测试命令、退出码、摘要和观察到的失败。
- `post_test`：patch 后测试命令、退出码和摘要。
- `official_harness`：是否需要官方 harness、是否 resolved、source type 和 benchmark instance id。

设计边界：

- `verification_evidence` 是证据层，不直接决定验收结论。
- `verifier_report` 是判断层，基于运行状态和证据给出 `verification_level` / `risk_level` / `accepted`。
- `accepted_final_status` 是产品层状态，用于用户展示和人工验收。

预期效果：

- 用户不只看到“是否成功”，还能看到成功判断基于哪些证据。
- SWE-bench Lite local smoke 可以明确携带 `official_harness.required=true`，避免被误解为 official resolved。
- 未来引入 LLM Verifier Agent 时，可以优先读取 `verification_evidence`，不必重新解析完整 trace。
- 后续评测可以按 evidence 维度统计：有多少 run 有 pre-fail/post-pass，有多少只是 weak/static verification，有多少需要官方 harness。

## Agent Core Refocus v13：Evidence-aware Verifier Gate

### 为什么进入 v13

v11.2 已经把验证依据结构化为 `verification_evidence`，但如果 evidence 只是被记录下来，verifier 仍然停留在“展示证据”的层面。v13 的目标是让 verifier 真正消费 evidence，给出证据质量分级。

这一步仍然不增加 LLM 调用，也不增加工具调用；它只把已有运行事实转化为更明确的质量判断。

新增输出：

- `evidence_quality`：`strong` / `partial` / `weak` / `missing`
- `missing_evidence`：缺失的关键证据，例如 `pre_test_failure`、`post_test_success`、`full_verification`、`official_harness`

当前判定规则：

| 场景 | evidence_quality | 说明 |
| --- | --- | --- |
| 有 patch、pre-test 失败、post-test 通过、full verification | `strong` | 最可信的本地验证证据 |
| targeted verification 通过，但没有 full verification | `partial` | 有局部证据，但不能产品层接受 |
| SWE-bench Lite local smoke 通过，但缺 official harness | `partial` | 本地 smoke 不能等同 official resolved |
| weak/static verification | `weak` | 只能说明 patch 未明显失败，不能证明修复可信 |
| 无 patch 或无 post-test 证据 | `missing` | 缺少基本验证证据 |

预期效果：

- verifier 不再只输出结论，还会说明证据是否足够强。
- 用户可以区分 `success` 背后的证据质量。
- 后续可以把 `evidence_quality` 作为产品 UI、评测统计和 Verifier Agent 的统一输入。
- 进一步削弱“agent 自称成功”的问题，把成功判断下沉到确定性 gate。

## v14：Verification Quality 实际运行验证

### 验证目的

v11.2 / v13 已经实现了 `verification_evidence`、`evidence_quality` 和 `missing_evidence`。v14 不继续加新机制，而是用实际 run 验证这些字段是否真的能区分“模型任务完成”和“产品层可信完成”。

### 样本结果

| 样本 | Run ID | final_status | accepted_final_status | verification_level | evidence_quality | missing_evidence | LLM 调用 | Token | 工具调用 | 结论 |
| --- | --- | --- | --- | --- | --- | --- | ---: | ---: | ---: | --- |
| `task_010` | `run_20260626T082311707092Z_3322` | `success` | `accepted_success` | `full_verification_success` | `strong` | none | 5 | 18,603 | 8 | pre-fail / post-pass / full verification 齐全，可作为可信成功 |
| `swe_lite_pydicom_pydicom_1139` | `run_20260626T082356497374Z_0173` | `success` | `local_smoke_success` | `local_smoke_success` | `partial` | `official_harness` | 13 | 70,045 | 14 | 本地 smoke 成功，但明确不能等同 official resolved |
| `v14_weak_static_demo` | `run_20260626T082531148610Z_8521` | `incomplete` | `not_accepted` | `none` | `missing` | `patch, post_test, full_verification` | 16 | 70,546 | 20 | 无 patch / 无 post-test 时不会误报成功，直接标记证据缺失 |

### 定性结论

v13 的证据质量层在实际 run 中表现符合预期：

- 普通 semi-real full verification 成功被标为 `strong`，且 `accepted_final_status=accepted_success`。
- SWE-bench Lite 本地 smoke 即使 `final_status=success`，也被降为 `local_smoke_success`，并通过 `missing_evidence=official_harness` 说明缺少官方验证。
- 没有 patch 和 post-test 的用户入口样本被标为 `missing`，避免把失败或无证据运行包装成成功。

这说明 Verification Quality Layer 的价值已经从“字段存在”进入“实际区分结果质量”：用户可以通过 `accepted_final_status + evidence_quality + missing_evidence` 快速判断一次 run 是否真的可接受。

## v15：CLI Verification Quality 输出

### 为什么做 v15

v14 证明了 `accepted_final_status + evidence_quality + missing_evidence` 在实际 run 中有区分能力。但如果用户需要打开 `result.json` 才能看到这些字段，产品体验仍然偏实验化。

v15 的目标是把验证质量直接放到 CLI summary 中，让用户运行 `repair_bug.py` 或 `run_issue_agent.py` 后立刻看到“这次修复是否可接受、证据强度如何、缺少什么证据”。

### 实现范围

- 新增统一 CLI summary helper：`build_verification_summary_fields()`。
- `scripts/run_issue_agent.py` 输出：
  - `accepted_final_status`
  - `verification_strength`
  - `verification_level`
  - `evidence_quality`
  - `missing_evidence`
  - `verifier_accepted`
  - `risk_level`
  - `evidence_scope`
  - `evidence_official_harness_required`
- `scripts/repair_bug.py` 复用同一 helper，并补充 `missing_evidence`。
- README 用户入口补充 CLI 输出说明。

预期效果：

- 用户不需要理解内部 JSON，就能判断 run 是否可信。
- SWE-bench local smoke、weak/static verification、missing evidence 会直接暴露在终端输出中。
- 为 v16 的 Verifier Agent 输出打基础：未来 Verifier Agent 的结果可以继续通过同一 summary helper 呈现。
