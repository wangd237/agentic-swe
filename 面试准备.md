## Q1: python_repl 的"受控"具体是如何实现的？安全边界在哪里？

**面试官追问背景**：应聘者提到 agent 有一个"受控的 python_repl"，面试官质疑如果 agent 写出 `import os; os.system("rm -rf /")` 或读 `.env` 外发 API key，系统如何拦截。

**应聘者回答要点**：

**四层纵深防御**：

| 层 | 机制 | 文件位置 | 拦截能力 |
|---|---|---|---|
| 第一层 | 文本预处理：拒绝 `;`、`import`、`\n`、`__`、超 500 字符 | `app/tools/python_repl.py:89-101` | 拒绝多语句、import 关键词、dunder、换行 |
| 第二层 | AST 节点白名单审查（`_SafetyValidator`）：拒绝所有 import 节点、非白名单函数调用、lambda、推导式、海象运算符 | `app/tools/python_repl.py:41-86` | 封死 `__class__.__bases__[0].__subclasses__()` 沙箱逃逸链 |
| 第三层 | 空 builtins 执行环境：`eval(compiled, {"__builtins__": {}}, SAFE_GLOBALS)` — `open`、`exec`、`__import__` 全不可用，SAFE_GLOBALS 只有 12 个安全函数 | `app/tools/python_repl.py:104-109` | `open()` 报 `NameError` |
| 第四层 | 结果截断：1200 字符上限 | `app/tools/python_repl.py:116-120` | 防 context window 膨胀 |

**架构层补充防线**：
- Agent 运行在 git workspace 副本中，所有工具（包括 `read_file`）只能访问 workspace，读不到本机 `.env`
- `write_file` 有文件名黑名单（`debug.py`/`tmp.py`/`scratch.py`/`probe.py`），返回 `scratch_file_not_allowed`（`app/agent/tool_executor.py:21-26`）

**实际价值**：`task_048` 中 agent 不再靠脑补 `Version.base_version` 行为，而是通过 `python_repl` 查询真实结果 → 发现 `base_version` 会丢掉 prerelease/dev → 转向正确的 `.public` 属性 → 成功修复。

**面试官追问**：workspace 隔离是 `cp -r` 还是命名空间级别？如果 `run_tests` 里跑了 fork bomb 或恶意测试代码怎么办？有超时和资源限制吗？

**应聘者回答要点**：

## Q2: `final_status` 的判定逻辑 —— success 是怎么判的？

**面试官要求**：讲清楚真实代码里的状态机，别画流程图；如果模型写完文件后直接停了怎么办？

**应聘者回答要点**：

**三个核心状态变量**（`app/agent/llm_agent.py:521-531`）：

```python
workspace_generation = 0     # 每次成功 write/edit/undo 后 +1
verified_generation = None   # 最近一次 run_tests 时的 generation 快照
last_test_exit_code = None   # 最近一次 run_tests 的退出码
pending_auto_verification = False  # 存在未验证的新写入
```

**状态流转**：
- 写操作发生 → `workspace_generation += 1`，`pending_auto_verification = True`，`verified_generation` 不变 → 产生"代差"
- `run_tests` 被调用 → `verified_generation = workspace_generation` → 追平代差
- 如果模型没跑测试就停了 → `pending_auto_verification` 触发系统自动执行 `show_diff` + `run_tests`，结果回喂模型继续决策（第 731-784 行）

**判定函数 `_classify_final_state()` 的优先级树**（第 424-454 行）：

```
success 条件（三重 AND，缺一不可）：
  ① patch_applied = True      （workspace 相对原始 repo 有实际 diff）
  ② last_test_exit_code == 0  （最新一次测试通过）
  ③ verified_generation == workspace_generation  （没有未验证的写入代）

failure 分支（按优先级降序）：
  1. model_reported_incomplete  → "incomplete" / "task_incomplete"
  2. no_tool_calls              → "stopped_without_verification" / "no_tool_calls"
  3. max_iterations_reached     → "incomplete" / "max_iterations"
  4. not patch_applied          → "incomplete" / "no_patch"
  5. exit_code != 0             → "incomplete" / "failed_tests"
  6. no tests run               → "incomplete" / "no_tests_run"
  7. verified != workspace      → "incomplete" / "unverified_patch"
```

**真实案例验证**：
- `task_010` 成功路径：`edit_file`(gen=1) → `run_tests`(verified=1, exit=0) → `show_diff` → 模型总结 → triple-AND 全满足 → `"success"`
- `task_132` 早期被拒：测试绿但无写入 → `patch_applied=False` → 走到优先级 4 的 `"no_patch"` 分支 → `"incomplete"`，没有误报 success
- 防"先测后改"：若先测再改再停 → `verified_generation(2) != workspace_generation(3)` → `"unverified_patch"`

**面试官简评**：通过。应聘者对 `_classify_final_state` 的三重 AND 条件优先级有精确理解，能反过来用代码逻辑反推业务场景。

## Q3: frozen_40 的构成、策略版本门禁、与 flaky 判定

**面试官追问背景**：应聘者在自我介绍中提到"冻结集 frozen_40 连续 8 个策略版本无回归"。面试官质疑：(1) 这 40 条任务是随机采样还是精选的容易题？ (2) 策略迭代 71 个版本，每次发版前跑什么基准？如果 frozen_40 无回归但全 66 条退化，系统能捕捉到吗？ (3) task_132 的 no_patch→success 用例如果在 frozen set 里，"连续无回归"用哪次指标？

**应聘者回答要点**：

**frozen_40 的构成**：
- 非随机采样也非 cherry-pick，是从正式 66 条全集中按 task ID 连续段截取（task_006 至 task_081，偶数为主）
- 包含硬骨头（task_030 inline table 格式、task_048 packaging 版本语义），也包含简单题（task_010），自然覆盖难度谱
- **关键澄清**：压力测试中的 task_030/task_048 失败发生在 LLM agent（DeepSeek）上，而 frozen_40 streak 追踪的是**规则版 baseline**（improved_v 系列策略），两个系统不同。规则 baseline 对这类任务有确定性 patch 模板，稳定通过。

**Streak 门禁机制**：
- 合格版本门槛（`analyze_benchmark_maturity.py`）：成功率 ≥ 95%、测试通过率 ≥ 95%、时延 ≤ baseline(v32) 的 103%
- Streak 找的是最长连续合格版本序列——中间断一个版本就重新计数
- **三层防回归漏斗**：
  1. frozen_40 streak — 第一道快速哨兵
  2. `stability_recheck` — 同一策略复跑 N 次验证 flaky 稳定性
  3. `compare_evals` — baseline vs improved 全维度逐指标 delta 对比
- 因此 "frozen_40 无回归但全集退化" 理论上可以被 compare_evals 的 taxonomy 标签变化和 partial_fix_rate 下降捕捉到

**flaky 处理与 task_132**：
- `task_132` 不在 frozen_40 中（编号远超 081），在压力测试集中
- 同版本多次 eval run 使用"最新文件名覆盖"策略，非显式 flaky 处理
- `stability_recheck` 有 `functional_consistent` 字段（所有复跑成功率和测试通过率完全一致才算一致），但这是独立路径，不参与 streak 计算

## Q3 追问: frozen_40 streak 的"无回归"到底衡量什么？如果是聚合成功率 ≥95%，能否掩盖逐题退化？

**面试官追问**：你承认 streak 用的是聚合门禁（成功率 ≥95%）。这引申出一个问题——一个版本能把 task_010 和 task_024 修好，但同时把 task_048 和 task_056 搞坏，成功率一样是 95%，streak 判定为"无回归"。这种设计是否存在结构性问题？

**应聘者回答要点**（Postmortem 式自评）：

**当前架构的三条核心局限**：

1. **聚合门禁 ≠ 逐题回归保护**。Streak 看的是 `success_rate ≥ 95%`，不是 per-task 状态差分。理论上可以出现"失败面孔完全换人但成功率不变"的暗回归，而 streak 完全察觉不到。需要的是 `compare_evals` 层面的逐题 `final_status` 差分矩阵。

2. **Streak 基于单次评测快照，不经过稳定性验证**。`stability_recheck` 的复跑机制可以检测 flaky，但它和 streak 计算是两条独立的代码路径——streak 只看单次 eval 结果文件的最新编号覆盖。一次环境波动导致的 timeout 可能打断 streak，一次 flaky 成功也可能被计入。

3. **Baseline 锚点漂移**。时延门槛（≤ v32 的 103%）锚定在 `improved_v32` 这个历史上任意选的版本上。随着策略从 v32 迭代到 v71，如果某次优化天然增加时延（如更复杂的正则匹配），即使功能无退化也会被踢出合格版本。阈值应该是一个持续更新的滑动窗口而非固定锚点。

**如果只改一件事**：在 `compare_evals` 中加入 per-task `final_status` 差分表，回归的定义从"聚合成功率下降"升级为"任何一条任务从 success 变为非 success"。

**面试官简评**：通过。应聘者没有为聚合成功率设计辩护，诚实承认 per-task 回归差分不在当前体系中；三条 postmortem 条目具体可落地。建议补充 v32 vs v71 的具体时延数据使锚点漂移问题更直观。

## Q4: 规则版 100% vs LLM agent 91.3%——既然规则版更强，为什么还要做 LLM agent？

**面试官追问背景**：应聘者自我介绍中提到"双轨结构"——规则版 baseline 作为对照。面试官注意到规则版成功率 100%、LLM agent 仅 91.3%，质疑为什么要费力做 LLM agent，规则版的上限在哪里？

**应聘者回答要点**：

**规则版的本质**：不是 AI 系统，而是策略版本化的查找表。71 个版本就是往表里加 71 条精确的字符串匹配→字符串替换规则。每条规则只对一种已知代码模式有效。

**核心区分——闭世界 vs 开世界**：66 条 benchmark 是跟规则表同步成长的——每当我们碰到一种新 bug 类型，就为它写一条规则，然后把它收进任务集。100% 意味着"为已知问题写的规则在已知问题上管用了"，是闭世界成就。

**LLM agent 能做而规则版永远做不了的事**（以 `task_048` 为例）：
- 规则版无法处理——要写出正确修复需要知道 `Version.base_version` 和 `Version.public` 到底返回什么，这超出了字符串替换的能力范畴
- LLM agent 的做法：`(1)` 读代码 → `(2)` 用 `.base_version` 做第一版→测试失败 → `(3)` 调用 `python_repl` 查询真实 API 行为 → `(4)` 发现 `base_version` 会清掉 dev/prerelease → `(5)` 换用 `.public` → 测试通过
- 关键差异：LLM agent 在不认识 `packaging.Version` API 的情况下，通过工具去运行时查询了真实行为，再把事实转化成修复逻辑

**补充案例——LLM agent 反而搞砸的任务**（`task_030`）：
- tomlkit inline table 缺 `, ` 分隔符的修复。规则版有精确的 `broken_body`→`fixed_body` 字符串替换，一把过
- LLM agent 定位正确但写多了两个 `\n`，测试失败后反复修改 13 轮 hit `max_iterations`
- 揭示的不对称：规则版"笨"得精确——不知道变通所以该精确的地方绝对精确；LLM agent 在需要像素级格式精度的任务上反而不如字符串替换稳定
- 压力测试后给 `run_tests` 加了 `output_excerpt` 功能，让模型能看到实际值 vs 期望值的文本差异，不是只看到一个 assertion failed

**面试官简评**：通过。用 `task_048` 解释"规则版进不了门"、用 `task_030` 解释"LLM agent 反而搞砸"——两个案例对称展示了双轨结构的互补性，说清楚了 100% 和 91.3% 背后的本质差异不是"谁更强"而是"不同能力谱的定位"。追问了 `task_030` 的失败分析是否有后续 `context_diff` 验证，但意思已到。

## Q5: Agent 循环内部机制 —— 工具调度并行、上下文压缩、反循环检测

**面试官追问背景**：面试官注意到应聘者多次提到"max_iterations"和"试探→失败→修正"的迭代模式，追问循环内部三个具体机制。

**应聘者回答要点**：

**① 工具并行调度**：

- LLM 单轮可返回多个工具调用，调度器按类型拆分为两拨：
  - **只读操作可并行**（`read_file`、`search_code`、`grep`、`show_diff`）：线程池同时跑，互不干扰
  - **写入/有副作用操作必须串行**（`write_file`、`edit_file`、`undo`、`run_tests`、`python_repl`）：分批串行执行，保证 git commit 链不乱和 workspace 状态一致
- 典型模式：一轮并行跑 3 个读操作 → 下一轮串行写入 → 再下一轮跑测试

**② 上下文压缩**：

- 策略：**留头（任务描述永远保留）+ 留尾（最近 3 条消息原样保留）+ 中间压缩成摘要**
- 压缩内容：中间每条消息提取关键事件信息——哪轮调了什么工具、核心结论、改了什么文件——压缩成事件列表而不是完整对话
- 已知代价：中间轮次的测试失败细节可能在压缩中丢失。为此给 `run_tests` 的 failure summary 额外保留空间，让关键断言信息即使跨越压缩也能被模型引用
- 触发点：模型生成响应后 + 工具结果回灌后，每次检测字符数是否超阈值

**③ 反循环检测**：

- 检测逻辑：跟踪最近三次成功写入操作，若三者满足——指向**同一文件**、且写入内容**高度相似**（old_string 完全相同，或新文本相似度 > 80%）——则判定为"写入死循环"
- 触发动作：注入一次性提醒，告诉模型"退一步、重新读代码验证核心假设、优先用 python_repl 查实际行为"
- **诚实承认盲区**：`task_030` 的 13 轮循环没有触发此检测——它每次修改方向不同（改换行→改逗号→改格式拼接），在操作级别上不重复，但在概念级别上在兜圈。差距在于：当前机制能抓操作级重复，抓不住概念级兜圈
- 改进思路：检测最近 N 轮 `run_tests` 是否都指向同一道测试的同一个失败、且失败信息一直没变——若成立，说明不管怎么改测试始终在同一件事上报错，也应触发换思路提醒

**面试官简评**：通过。并行分拨逻辑正确；上下文压缩的"留头留尾压中间"务实，且应聘者清楚知道"压缩丢细节"的代价并针对性给 `run_tests` 做特殊处理；反循环检测能诚实指出 `task_030` 没被抓住的盲区，并能区分"操作级循环"和"概念级兜圈"。

## Q6: 从 v1 到 v71 到底迭代了什么？规则冲突怎么处理？

**面试官追问背景**：应聘者反复提到 71 个策略版本。面试官质疑：(1) 版本里装了什么——调参、规则、还是 prompt？(2) 如果主要是加字符串匹配规则，叫"71 个版本"是否夸大？(3) 规则膨胀后会不会互相打架？

**应聘者回答要点**：

**版本内容构成**：约 69 个版本是各加一条新的字符串匹配修复规则；约 2 个版本是 pytest 运行时参数调整（如开关 `unraisableexception` / `threadexception` 插件）。没有一个版本涉及 prompt 改动——prompt 是 LLM agent 轨的事。

**对"夸大"的坦率承认**：如果诚实地说，这不是 71 次算法迭代——没有架构重构、没有策略范式变化，就是一张规则表在不断变长。叫"版本号"的实际用途是**可追溯性**——frozen_40 上哪个策略出了问题，能立刻定位到"improved_v56 加进来的那条规则"。更好的叫法可能是"规则快照"或"规则集"。

**规则冲突消解**（fall-through 链）：
- 所有规则串成一条链，最新版本先匹配，先到先得
- 每条规则双重保护：(a) 匹配前检查目标代码模式是否真的存在；(b) 匹配后检查是否已被修改过，若已有修复后的代码则跳过
- 冲突被链式顺序消解——v62 先命中后，v45 轮到时会发现目标代码块已不存在，自动跳过
- **伴生风险**：如果 v62 的规则改错了，后面所有规则都没有修正机会（它们是字符串匹配，不是推理）。因此每加一条新规则必须跑 frozen_40 回归门禁——streak 的实际用途正在于此

**面试官简评**：通过。应聘者直接承认"71 这个数字有水分"，愿用"规则快照"替代"策略版本"。回答中揭示了一个更重要的点：每条新规则要过 frozen_40 门禁，这才是真正的工程纪律而非版本编号。面试官追问了一个边界 case——新版本 v71 的规则如果依赖的代码块已被 v62 的规则改掉了怎么办？应聘者未展开但 fall-through 链的"先到先得"逻辑已说明了这一问题存在的可能性。

## Q7: Tool Call 纠错机制 & 工具 Schema 设计哲学

**面试官追问背景**：面试官聚焦 agent 技术核心，追问：(1) LLM 产出不合法 tool call（参数缺失、JSON 解析失败、幻觉工具名）时系统如何处理？(2) 为什么有 10 个工具，特别是 `grep` 和 `search_code` 高度重叠为什么不合并？

**应聘者回答要点**：

**① Tool Call 纠错的三种场景**：

| 场景 | 处理方式 | 关键设计选择 |
|---|---|---|
| JSON 参数解析失败 | 静默降级为空字典 `{}`，不抛异常，工具以缺省参数执行 | 与其中断 run，不如让工具返回错误让模型从反馈中学习 |
| 调了不存在的工具 | `ToolExecutor` 返回 `ok: false, error_type: "unknown_tool"`，消息回喂给模型 | 模型看到错误后在下一轮换正确工具名 |
| 参数缺失/非法值 | 各工具自身的防御逻辑（如 `grep` 空 pattern → `invalid_pattern`，`read_file` 路径不存在 → `not_found`） | 错误包含 `error_type` + `message`，构成自然纠错信号 |

**核心哲学**：纠错不是"重试机制"，是"对话流"。模型犯错 → 工具返回错误 → 错误消息加入对话历史 → 模型下轮修正。机制生效前提是 `max_iterations` 足够（默认 13 轮），一个"犯错→修正→再试"通常只占 2-3 轮。

**② grep 与 search_code 不合并的理由**：
- 底层都是遍历文件逐行匹配，但给 LLM 的认知接口不同
- `search_code`：简单子字符串匹配，"找到所有提到 base_version 的地方"——不需要正则思维
- `grep`：正则匹配 + `glob` 文件过滤 + `max_results` 截断，"找到所有以 `def contains` 开头的函数定义"——精确结构匹配
- 分开是因为 LLM 规划时两种场景的**心理模型不同**。若合并成一个工具加 `mode` 参数，会增加选择负担导致选错模式
- "10 个不是设计出来的，是长出来的"——从最初 5 个逐步演进补全，每加一个新工具的标准是：**现有工具组合能否覆盖此场景**？能就不加

**③ 工具数量过多 vs 过少的经验**：
- 过多：曾讨论加 `search_tests`，放弃——`search_code` + `glob` 即可覆盖
- 过少：`python_repl` 加上之前，`task_048` 的 agent 只能脑补 API 语义反复猜错——工具不足导致"在黑暗中摸索"

**面试官简评**：通过。纠错回答将三种异常场景逐一展开，特别是 JSON 解析失败"静默降级而非抛异常"的设计选择体现了务实的 tradeoff。grep/search_code 用"心理模型"角度（而非代码复用/性能）解释不合并的理由更有说服力。"工具是长出来的不是设计出来的"放在面试里比"深思熟虑的架构设计"更可信。面试官追问了一个未展开的问题——是否有日志数据分析 LLM 更偏好哪个搜索工具，若 90% 只用其中一个则另一个实际可能是冗余的。

## Q8: Prompt Engineering / Context Engineering / Harness / Loop Engineering 的边界

**面试官追问背景**：面试官要求应聘者区分四个易混概念，并指出哪个最易被忽视但影响最大。

**应聘者回答要点**：

- **Prompt Engineering**：模型看到的文字指令—— system prompt（角色、工具、约束）和 user prompt（任务描述、issue、成功标准）。是最表层的东西。
- **Context Engineering**：模型执行过程中能"看到"的所有信息——工具返回结果的截断策略、回喂格式、压缩时机与粒度。Prompt 是出发前的地图，context 是行军途中每一步的路况。
- **Harness**：agent 运行的"操作系统"——workspace 隔离、git checkpoint、路径边界约束、工具执行环境。不产生决策，但所有决策依赖它。
- **Loop Engineering**：控制流——何时继续、何时停、写操作后是否自动补验证、重复写入怎么检测、上下文何时压缩。决定了执行骨架。

**核心判断——Context Engineering 最易被忽视但影响最大**：

以 `task_048`（packaging 版本语义修复）为例：早期 13 轮失败不是因为模型不聪明，而是测试失败后模型只能看到 `AssertionError: False is not True`——没有实际值、没有期望值、没有差异对比。这就是 context engineering 的缺口——失败信息没给够，模型只能靠猜。后来给测试失败摘要加了失败输出截取片段，模型能看到"我返回了什么 vs 期望是什么"的对比，才能收敛到正确方向。这个改进不是改 prompt 也不是改 loop，是让测试失败消息告诉模型更多东西——context engineering 的范畴。

**追问：有没有"信息太多"的问题？怎么验证压缩没产生退化？**

- 信息太多：大文件读取被截断到合理长度，搜索结果限 50 条，文件列表限 8000 字符。但也有代价——有一次 bug 出在文件后半段，恰好被截断阈值切掉了，模型完全看不到问题代码。
- 压缩质量验证：没有做系统 A/B 实验。粗糙的观测是看压缩后模型是"收敛"（继续原方向修复成功）还是"发散"（开始乱读无关文件）。但没法区分"压缩丢掉了关键信息"和"模型本来就在碰运气"。
- 若补实验：frozen set 上跑压缩 vs 不压缩两组对比，统计上无明显差异则代价可接受；若出现系统性退化则需精细化策略。

**面试官简评**：通过。能把四个概念的边界讲清楚，且用 task_048 的 output_excerpt 改进说清了 context engineering 为什么最关键的判断。追问部分诚实承认了没有系统实验但知道怎么设计。

## Q9: Agent 范式对比（ReAct / Pipeline / Plan-then-Execute）& Claude Code 对比 & Skills vs Tool Calling

**面试官追问背景**：面试官不再只问项目内部，延伸到更广的 agent 生态——要求对比多种范式、对比 Claude Code 与自己项目的差异、追问 Skills 和渐进式披露的本质。

**应聘者回答要点**：

**① Agent 范式谱系（按"自主权"从刚性到柔性）**：

| 范式 | 特点 | 适用场景 |
|---|---|---|
| 纯 Workflow / Pipeline | 每步预先编排，绝对可控 | 已知问题模式，无需应变 |
| ReAct（思考→行动→观察→循环） | 每步基于当前观察动态决策 | 需要中途调整方向的修复任务 |
| Plan-then-Execute | 先完整计划，再按部就班执行 | 确定性高、不需要中途纠偏的场景 |

**选型时为何没选 Plan-then-Execute**：(1) 任务集中很多需要"读完代码发现不在最初猜测的位置"的中途调整，Plan-then-Execute 不友好；(2) 当时用的模型首次规划容易漏关键步骤——与其信它一次性做对计划，不如让它每步可纠偏，ReAct 更容错。

**Multi-Agent**：了解但项目目前没用——当前任务复杂度单 agent + 工具调用够用。如果扩展至跨文件重构等复杂场景，multi-agent 有优势。

**② Claude Code vs 自己项目的本质差异**：

- **Claude Code**：追求"一次性成功率"——丰富的工具、交互式对话、允许用户打断矫正。不在乎能不能复现。
- **自己的项目**：追求"可复现的评测可信度"——每次 run 必须落盘完整轨迹、成功判定不能靠模型口头说、必须靠测试结果加 generation 验证。
- 核心差异：**"让一个用户满意"vs"让一个评测结果可信"**——两种目标导致的工程设计取舍几乎是相反的。

**③ Skills vs Tool Calling 的区分**：

- **Tool Calling** 扩展的是"行动空间"（能做什么）——每个工具是一只手
- **Skills** 扩展的是"方法空间"（怎么思考该做什么）——封装了"在什么情况下、按什么顺序、用什么标准"去调用哪些工具
- 类比：**菜刀和锅是 tool，菜谱是 skill**。Tool 让你能切菜炒菜，Skill 告诉你"宫保鸡丁应该先腌肉、再炸花生、最后调味"
- 自己的项目目前只有 tool 没有 skill——模型每次都从零编"菜谱"，这是陷入低效试错循环的根因之一

**④ 渐进式披露在项目中的应用（诚实承认未做）**：

- 现状：一次性把 system prompt + 10 个工具定义全灌入每条请求，不管当前轮是否用得上
- 为什么：10 个工具、66 条任务的规模下，全量灌入的浪费还不致命，简单先行
- 天花板：工具数从 10→50→200 时，全量灌入不可持续。渐进式披露那时是刚需——不仅工具按需注入，专项知识也应场景绑定
- 这本质上就是 Skills 的用武之地——把"什么时候该给模型什么信息"从"永远全量灌入"升级为"按需披露"

**面试官简评**：通过。范式对比放在"自主权谱系"上且选型理由基于实际任务特征而非泛泛列举。Claude Code 对比抓住了"设计目标不同"这个本质（一次性成功率 vs 可复现评测），不是功能清单级比较。"菜刀和菜谱"的类比精准区分了 Skills 和 Tool Calling。渐进式披露坦白了"没做"但描绘了"如果做会怎么做"的思路——知道一个概念名词不叫理解，知道它在什么条件下变得不可或缺、怎么落地，才叫理解。

## Q10: Vibecoding 与 Engineering 的分界线 & 人的判断能否编码进 Agent Harness

**面试官追问背景**：应聘者主动坦白项目全程 vibecoding。面试官追问：(1) vibe 和 engineering 的分界线在哪——AI 写了代码，你具体做了什么让系统没有退化成不可维护的浆糊？(2) vibecoding 经历和 agent 开发有没有关联，还是只是趣闻？深度追问：从 vibecoding 里积累的"人的判断"，有多少能编码进 harness，有多少编码不进去只能靠人留在循环里？

**应聘者回答要点**：

**① Vibecoding 里的三重角色**：
- 定义验收标准（"什么叫对、什么叫错"是我定的，不是 AI 定的）
- 审查关键路径（状态机、安全边界、判定逻辑——出错后果严重的地方必须人审）
- 打回不靠谱的输出（AI 是写代码的手，人是质量门禁）

**② 亲手把关的具体案例**：

*案例一——三重 AND 时序漏洞*：AI 最初写的成功判定是"模型跑了测试、测试过了就标 success"。应聘者审查时发现时序漏洞——如果模型先跑测试通过、再改文件、但不重新跑测试，系统也会标 success。亲手改成三重 AND（有 patch + 测试通过 + 验证代际匹配），后来真的阻止了一次"测试全绿但没有 patch"的误报。

*案例二——python_repl 沙箱的 AST 层*：AI 最初只做了文本层过滤（拒绝 import 关键词和分号）。应聘者意识到 `getattr(__builtins__, 'ex' + 'ec')` 在文本层面完全合法但在语义层可拿执行权限，亲手加了 AST 白名单审查——在语法树层面拦截所有非白名单节点。如果没加这层，"沙箱就是纸糊的"。

**③ Vibecoding 与 Agent 开发的关系——不是趣闻，是预演**：
- 两者面临同构问题：信息不对等、上下文有限、执行者看不到全局
- 在 vibecoding 里积累的判断力——知道什么时候该信任输出、什么时候必须介入——在设计 agent 系统时会反复用到
- 核心主张："如果我能管好一个写代码的 AI，那我能不能设计一套系统，管好一个修 bug 的 AI？"——两个问题在工程上高度同构

**④ 能编码 vs 编不进去——两个案例的对称对比**：

| 类型 | 案例 | 本质 | 编码方式 |
|---|---|---|---|
| 能编码 | 三重 AND 代际验证 | 时序关系——"A 发生之后 B 有没有发生" | 两个计数器差值检测，机器比人更擅长跟踪 |
| 编不进去 | 概念级兜圈检测 | 语义理解——"模型是不是在假装懂了" | 操作级循环检测能抓重复写入，抓不住换了方向但仍在同一误解外围试探 |

**⑤ "编不进去"时的应对策略——绕到上游而非硬堆规则**：
- `task_030` 中 agent 13 轮未收敛——每次改法不同（换行→逗号→格式拼接），操作级检测未触发，但概念级在兜圈
- 没有硬编码更复杂的检测规则，而是改善 context：给 `run_tests` 加了输出截取片段，让模型看到实际值 vs 期望值的对比
- 哲学：降低兜圈概率（改善输入）比死磕兜圈检测（硬堆规则）更可行
- 承认边界：有些判断真的没法编码时，设计上应留出人类介入的接口——很多生产级 agent 选"人留在循环里"不是懒，是认知边界

**面试官简评**：通过。两个案例的对称结构（"编进去了"+"编不进去"）比单纯讲成功案例更有说服力。关键贡献是把"能编码"的判断本质提炼为**状态关系**（时序、计数、存在性），"编不进去"的本质提炼为**语义理解缺失的检测**——给出了可操作的分类标准而非笼统抱怨。面对"编不进去"时选择"绕到上游做 context engineering"而非"在下游硬堆规则"的策略方向感正确。未展开的点：项目目前有没有人机交互接口（全自动还是半自动），可后续追问。

## Q11: HITL 该插在哪？& Vibecoding 带去的可迁移经验

**面试官追问背景**：承接 Q10 "编不进去的判断只能靠人"的结论，面试官追问：如果要给一个全自动 agent pipeline 加人机交互节点，应该插在什么位置？入职后第一件事就是给现有 agent 系统加人机交互界面，从这个项目带过去的哪条经验最有帮助？

**应聘者回答要点**：

**① 节点位置——不选"每个任务之后"也不选"每轮中间"，选"判定树走到不确定叶子"**：

应聘者指出判定树的七个失败分支存在隐含分类：
- "确实失败了"（测试挂、没 patch、没跑过测试）→ agent 可以自己继续，不需要人
- "不知道该干什么了"（max_iterations 耗尽、模型自报 incomplete）→ 自动化走到尽头，但系统强行停了

人的介入应该只触发于后者。具象化：如果 agent 用完了 13 轮迭代、测试还没过、但已经产生了部分 patch → 弹出一个人，"这个任务 agent 做了这些修改，测试差这一道没过，你要接受部分修复、指新方向继续、还是标记放弃？"人花 30 秒做决策比系统直接判 incomplete 然后默默停掉，下游利用率更高。

**核心设计原则**：人不是流程瓶颈，人是不可编码判断的兜底开关。91.3% 全自动通过，剩下 8.7% 由人做最后一次判决。

**② 可迁移的核心经验——不是编码能力，是"observability 先行"**：

- *经验一——结构化 trace 本质上是信息设计*：给 agent 的每一步工具调用做结构化轨迹落盘（哪轮调了什么工具、传了什么参数、返回了什么、模型据此做了什么决策）。如果 trace 只是原始 JSON 序列化，人不愿意读。trace 的设计本质是提取人需要的那几个维度、用最少理解成本呈现。

- *经验二——agent 和人需要看到的信息不一样*：测试失败的 output_excerpt 对 agent 是"看到差距以修正方向"，对人则是另一套信息需求。当给人机交互界面加信息展示时，必须同时回答两个问题——"agent 需要看到什么？人需要看到什么？"两个答案不一样，但必须同时服务好。

- *经验三——区分两种 HITL 范式*：一种是"agent 不行了喊人"（agent 主动调用 help 工具），另一种是"人本来就在观察，关键时刻可以插手"（harness 层设计的 trace 让人能在被通知后三秒内建立上下文）。应聘者明确偏好后者——"后者要求人在收到通知之前，context 已经建立好了，不能等被通知之后从头看 trace。trace 的信息设计正是这件事的基础。"

**面试官简评**：通过。应聘者否掉了"每任务审批"和"每轮打断"两个常规选项，提出触发点应定位在判定树中"不确定/证据不足"的叶子节点——这个粒度定位既不是过度谨慎也不是过度乐观。关键贡献是把 HITL 拆成"agent 不行了喊人"和"人本来就 ready，关键时刻插手"两种范式，且明确偏好后者并指出后者的前提是"人在收通知前已有足够 context"。未展开但值得追问的点：怎么让一个没看过全程的人在 30 秒内建立起对当前 run 状态的足够理解——这需要 trace 的信息压缩和呈现设计，应聘者提到了但未具体展开。

## Q12: Agent Run 的"30 秒摘要"协议设计 & Meta-Evaluator 的角色

**面试官追问背景**：承接 Q11 "人收到通知后必须在 30 秒内建立上下文"的未展开话题，面试官将问题升级为系统设计题：给 agent 每次 run 设计一个"30 秒摘要"，让人不用看全程就能决定要不要插手——这个摘要必须包含哪三到五个维度？每个维度怎么压缩？以及这个摘要能不能由另一个模型来写？

**应聘者回答要点**：

**① 五维摘要协议（按读序排列，支持提前退出）**：

| 维度 | 时间预算 | 内容 | 压缩策略 |
|---|---|---|---|
| 判决距离 | 3 秒 | 判定树的当前叶子节点标签 + partial_patch yes/no | 七个叶子节点枚举值本身就是压缩，不需推理 |
| 收敛轨迹 | 5 秒 | 最后三次测试结果的变化方向（↗收敛 / →原地踏步 / ↘发散 / ○未跑） | 只取 exit code 序列方向，抛弃具体值、堆栈、测试名 |
| 阻塞类型 | 5 秒 | 四个桶之一：API 语义缺失 / 格式精度不足 / 方向性错误 / 环境超时 | 由规则从 trace 自动化提取（如 `max_iterations` + exit code 不变 + patch 存在 → 格式精度不足） |
| 可操作动作 | 5 秒 | 不超过三个选项：[接受] [提示] [重跑] [放弃] | 由判决距离 + 阻塞类型联动派生 |
| 投入产出 | 12 秒 | 已用迭代 N/M / 已用时间 / 产生了 patch: yes(N files, K lines) | 纯数字，不帮助决策但帮助判断"值不值得花时间" |

**② 读序设计的隐藏原则**：人读到任何一个维度可以随时停止——判决距离如果是 success 就不需要人、读完收敛轨迹如果是"收敛中"就再给它几轮、读到阻塞类型如果是环境超时就重跑。维度排布不是并列，是**支持提前退出的树状决策链**。

**③ "让另一个模型写摘要"的约束**：
- 摘要模型应该是独立的 meta-evaluator，不是"正在修 bug 的那个 agent"——当事人和裁判必须分离。修 bug 的 agent 有让它失败的认知局限，让它总结自己为什么失败，它会重复那个让它失败的判断
- meta-evaluator 的输入必须是**已结构化的轨迹**（迭代轮次、工具调用序列、test exit code 序列、写入文件列表），不是原始对话——否则 meta-evaluator 本身也会遇到同样的 context window 问题。原始 run 不管多长，结构化事件流的大小和迭代轮次成正比，而非和 token 数成正比

**④ 三级分工模型**：
- 自动化提取做 80%（维一判决距离和维五投入产出直接从轨迹拿）
- 轻量分类规则做 ~15%（维二收敛方向、维三阻塞类型的多数 case）
- meta-evaluator 做边缘 case（维三中"方向性错误"vs"API 语义缺失"边界模糊时）
- 人做最后 5%（读摘要 + 选可操作动作）

**⑤ meta-evaluator 的输入/输出接口设计**：

输入（结构化事件流）：
```
iteration:4|tool:read_file|target:src/version.py|lines:1-50
iteration:4|tool:python_repl|query:Version("1.0.dev0").base_version|result:1.0
iteration:5|tool:edit_file|target:src/version.py|old_str:return .base_version|new_str:return .public
iteration:5|tool:run_tests|exit:0|duration:2.3s
```

输出：五维结构化摘要（维三的阻塞类型 + 维二收敛方向如需要 + 维四可操作动作列表）

**面试官简评**：通过。应聘者没有停留在"摘要很重要"的口号上，而是完整设计了一套可执行的五维协议。亮点：(1) 读序支持提前退出而不是并列罗列——这是 crash report 设计的核心原则，应聘者没明说但维度排布体现了它；(2) "收敛轨迹"只取方向不取绝对值，直接提炼为四个符号协议，取舍正确；(3) meta-evaluator 输入必须是"结构化轨迹"而非"原始对话"——避免了用第二个模型解决第一个模型的 context 问题的递归陷阱。

未展开但重要的点：
- 维四"可操作动作"的映射组合数量——面试官追问如果组合太多，是枚举规则还是交给 meta-evaluator 生成，应聘者未展开。这是从协议设计落地到实际实现的关键分岔口。
- 隐含问题：这套五维协议本质上就是一台"人脑的 API spec"——如果决策规则也能被自动化，HITL 在什么条件下可以退出循环？这是下一轮可追的方向。

## Q13: 评测可信度的三层分级 & Badcase 聚类 & "用模型审 patch"的设计约束

**面试官追问背景**：面试官从 Q12 的 HITL 可退出性延伸，追问评测体系的根本问题：(1) 你怎么证明"系统标了 success"是可信的？三重 AND 只防了"明明没修好但说修好了"，但防不了"修好了但引入了新 bug"；(2) 能不能用另一个模型做代码审查员？(3) 13 次失败能不能一分钟聚类？

**应聘者回答要点**：

**① 三层信任分级（核心框架）**：

| 层级 | 问题 | 当前系统 |
|---|---|---|
| L1: 协议正确性 | "系统说 success 是否意味着测试真的过了？" | ✅ 三重 AND 防了时序漏洞——gen 计数、测试退出码、代际匹配三个计数器交叉验证 |
| L2: 功能正确性 | "测试过了是否意味着 bug 真的修好了？" | ⚠️ 依赖测试用例质量——测试用例本身覆盖不足时，agent 可能修了表面症状而未触及根因，测试照样绿灯 |
| L3: 无副作用 | "修好这个 bug 的同时有没有引入新 bug？" | ❌ 完全没覆盖——patch 改了 A 文件的 B 函数，C 文件的调用方没被测试覆盖。这在自动修复领域是开放问题 |

应聘者坦承：当前系统只做到 L1。L2 不是 agent 的问题而是测试用例设计质量问题，但影响对 agent 的信任。L3 是结构性的——frozen_40 里如果恰好有跨文件依赖的回归 case 可能被 stability_recheck 捕捉到，但这是概率性的防御而非系统性的证明。

**主动证明 L3 的两个方向**：
- 方向一：运行目标仓库的**完整测试套件**，不只任务自带的那几条。但全量测试跑的代价是完全不同的量级，且有些测试依赖数据库、网络、特定 OS。目前没做。
- 方向二：用另一个模型做 code review（见下）。

**② "用另一个模型当代码审查员"的设计约束**：

应聘者提出关键前提：审查员不能只看 patch，必须看 **"patch + 收敛轨迹"**。

反例：task_030 的 patch 只加了一个逗号一个空格——如果只看 patch，审查员会给高分。但 patch 经历了 13 轮迭代才碰对——它本质上是"蒙对的"而非"理解的"。纯粹基于 patch 的审查会给低质量修复高置信度。

设计规则：
- agent 一把过（读代码→改一处→测试绿）→ 审查员给高置信度
- agent 改了 7 次、每次改不同方向、最后才绿 → 审查员标"低置信度：修复可能是碰对的"

两个未解决的实际问题：
- 审查模型和修 bug 的模型如果是同一个，会有相同的认知盲区——审查应使用不同模型或不同架构
- 审查模型需要完整 issue + repo context 才能判断 patch 合理性，每条审查的上下文成本不低

**③ Badcase 聚类的三维度框架**：

| 维度 | 内容 | 提取方式 |
|---|---|---|
| 阻塞阶段 | agent 停在"没找到 bug 位置" / "找到了但没修对" / "修了对的但没跑验证" | 从结构化 trace 直接查——检查最后一次 `run_tests` 前有无 `write_file`/`edit_file` |
| 失败面孔一致性 | 同一条任务多次失败 run 的 `final_status` 和测试失败信息是否相同 | 同任务横跨多次 run 对比——相同 → "系统性难题"；不同 → flaky |
| 根因粗略分类 | API 语义缺失 / 格式精度不足 / 方向性错误 / 环境超时 | 规则覆盖多数 case，边缘 case 需人工判断 |

**预判分布**（基于项目经验判断，非系统计算）：
- 格式精度不足 ~50%（知道改哪里但字符串差一点，反复改 hit max_iterations）
- 方向性错误 ~25%（修复策略本身错误，没触及真正的根因）
- API 语义缺失 ~15%（不认识库的 API 行为，没调用 `python_repl` 就硬猜）
- 环境/超时 ~10%

**诚实承认的差距**：从 trace 到三维度标签之间的**自动化映射未集成**为可一键调用的工具。零散规则存在，但没有 `--badcase-report` 这样的统一入口。根因："项目规模小（66 条、149 次跑），手动翻 trace 还能接受，没疼到非做不可。小规模单人的惯性——不是技术上做不了，是没把优先级提上去。"

**面试官简评**：通过。三层信任分级把"系统可信"分解为三个独立的可验证问题而非笼统判断——L1 用状态机、L2 靠测试质量、L3 是开放问题，三个答案不同且诚实。代码审查员的设计约束（"不能只看 patch，要看 patch + 收敛轨迹"）是 task_030 实战教训的直接应用，有具体反例支撑。badcase 聚类给出了可操作的三个维度而非泛泛的"失败原因多种多样"。

一个微妙的观察：应聘者对 badcase 回答了具体分布预估，但这些数字来自经验判断而非系统计算。他说"技术不难只是没疼到非做不可"——在小规模单人项目合理，但从团队协作角度，"可查询性"本身是一项工程能力，单人时不痛、团队协作时就晚了。这不是错误，但可以更有意识——把 badcase 查询从"手动翻"提升到"一键报告"不是一个锦上添花的优化，是可信评测体系的闭环最后一公里。

## Q14: "修好一类、搞坏另一类"的非对称效应 & 如何提前发现

**面试官追问背景**：承接 Q13 badcase 聚类后"总要修吧"的必然下一步——问应聘者调 prompt 或改策略时是否遇到过"修好 A 类 case 但 B 类本来能过的反而挂了"，以及怎么发现。

**应聘者回答要点**：

**① 具体案例——上下文压缩的非对称伤害**：

- 改动：压缩阈值从保守调激进（更早触发中间轮次摘要压缩）
- 现象：短链任务（3-4 轮修完）成功率不变。一条需要跨模块追踪多个文件的长链任务从 success → max_iterations
- 根因：压缩将中间轮次的搜索结果摘要化——保留了"搜索了 X 文件，找到 Y 函数"的事实，但丢失了"为什么这个函数可疑"的上下文集。模型后续需要引用前面线索时只能看到含糊摘要，被迫重新搜索，额外消耗迭代轮次
- 结论：压缩策略伤害有方向性——长链任务全受伤，短链基本免疫。这是结构性 tradeoff 而非随机波动

**② 发现途径——从"事后跑全集"到"手动加长链哨兵"**：

- 第一次发现：跑完 66 条全集对照 baseline 后才看到有一条之前能过的挂了——事后检测
- 意识到 frozen_40 的结构漏洞：按 task ID 连续段截取，不是按任务特征均匀分布。长链任务如果恰好不在 frozen_40 范围内，快速哨兵扫不到
- 临时补偿方案：每次改动后除了跑 frozen_40，额外手动挑两条已知的长链敏感任务单独跑——"不够优雅但有效"。有一次压缩调参 frozen_40 全绿但手动长链任务挂了一条，没等全量就止损了

**③ 如果重做的两个改进**：
- frozen set 不按 task ID 截取，而是**按任务特征分层采样**——至少保证每一层（短链/中链/长链、简单/中等/困难）在冻结集里都有代表
- 在 compare_evals 里加 **per-task final_status 差分表**——回归定义从"聚合成功率下降"升级为"任何一条任务从 success 变成非 success"。两件事加在一起，把"修好一类搞坏另一类"从跑完全集才发现变成哨兵阶段就能抓

**面试官简评**：通过。压缩案例精确到"长链受伤、短链免疫"的非对称效应而非笼统抱怨。手动挑长链任务的临时方案虽然朴素，但揭示了正确方向——frozen set 的采样维度应该匹配改动类型（改压缩 → 哨兵多放长链；改 tool schema → 哨兵多放需要那个 tool 的任务）。per-task 差分 + 特征分层采样的改进思路具体可落地。

## Q15: Vibecoding 中用过的工具/Skills & 对最终质量贡献最大的功能

**面试官追问背景**：面试官结合面经中的 vibecoding 实操问题，问应聘者在用 Claude Code 等工具做项目时用过哪些 Skills/Plugins，选择标准是什么，以及对最终质量贡献最大的功能。

**应聘者回答要点**：

**① 用最多的：code-review**：
- 不是在每次 commit 都跑，只在三种情况触发：改动涉及安全边界（沙箱、路径约束）、改动涉及状态机流转（成功判定逻辑）、大块重构之后
- 审查模型和修 bug 的 agent 不是同一个模型——"当事人和裁判必须分离"
- 选择性触发跟之前讲的 vibe/engineering 分界线是同一条原则——关键路径三道关，非关键路径可适当信任 AI

**② 刻意没用的：TDD**：
- 早期没用——因为接口还没稳定，AI 生成的第一版和第二版架构可能完全不同，这个阶段写测试第二天架构一改测试全废
- 中后期后悔——harness 稳定、接口定型后应该补测试。三重 AND 的时序漏洞如果在早期有单元测试，应该被测试抓出来而不是靠肉眼审查才发现
- 结论：vibecoding 和 TDD 有一道分水岭——接口稳定前 TDD 是负担，接口稳定后 TDD 是安全带。大部分人的问题是永远不跨过分水岭

**③ 贡献最大的不是生成功能，是验证约束**：
- vibecoding 最大的坑不是 AI 代码质量差——是"你合并了它写的代码之后，你没跑一遍就以为它做了它说它做的事"
- 立的铁律：每次 AI 完成一个功能点，必须在本地跑一遍验证——不是看代码像不像，是看行为对不对
- 这个习惯后来直接移植到 agent 系统设计里——三重 AND 的第三条（verified_generation == workspace_generation），本质上就是把"你跑过验证了吗？"这句追问从人问 AI 翻译成了机器问机器
- 核心判断：debug 和 review 类型的工具在价值上远大于生成类工具，因为 vibecoding 的核心矛盾不是"写得不够快"，是"写得不对但你不知道它不对"

**面试官简评**：通过。工具选择有取舍依据（不是全用也不是全不用），TDD 的"分水岭"判断务实——不是教条地支持或反对，而是给出了分阶段的条件。最有价值的观点是"贡献最大的是验证而非生成"——抓住了 vibecoding 的核心矛盾。

## Q16: Skills 体系设计 —— 技术形态、加载时机、触发机制

**面试官追问背景**：承接 Q9 中"菜刀和菜谱"的类比（Tool 是手，Skill 是经验），面试官追问：如果给项目 agent 加一套 Skill 体系，具体怎么设计？Skill 的技术形态是什么？和 Tool 的加载时机有什么区别？怎么让模型知道什么时候用哪个 Skill？

**应聘者回答要点**：

**① Skill 的技术形态 —— 三部分组成**：

- **触发条件**：什么情况下该用这个 Skill。不是让 agent 自己猜，而是由系统（harness 层）自动检测——因为触发信号需要横跨多轮轨迹才能感知（如"连续改了同一个文件三次但测试一直没过"），agent 在当前轮次看不到全局
- **实际内容**：一段很短的、针对当前场景的提示，帮 agent 从试错循环里跳出来。不是教它怎么修 bug，是给它一个更优的下一步动作建议
- **退出条件**：什么时候这个 Skill 不再适用了——比如测试通过了、或者用了三轮但没效果

**② 加载时机 —— 三选一，选中间方案**：

| 方案 | 做法 | 问题 |
|---|---|---|
| A. 全量常驻 | 所有 Skill 跟 system prompt 一起灌 | Skill 数量 ≥5 时 context 就爆了 |
| B. 工具调用式 | 给 agent 一个工具让它喊要哪个 Skill | agent 根本不知道有哪些 Skill 存在，不可能主动调——"菜谱锁在柜子里，厨师要喊开柜才能看，但厨师不知道柜子里有什么" |
| C. 轻量索引常驻 + 按需加载 | system prompt 里只有极短的 Skill 索引表（每个 Skill 一行：名称 + 一句话适用场景 + 触发信号），harness 检测到触发信号时自动把完整 Skill 内容注入到下一条消息 | 索引极轻不占 context，触发检测由系统做不依赖 agent 自主判断 |

**③ 为什么触发检测必须是 harness 层而不是 agent 自主喊**：
- "同一条测试连续失败 3 次"需要横跨多轮工具调用记录
- "Python 执行环境连续 2 轮未被调用"需要统计行为模式
- Agent 只能感知当前轮次它看到的东西，harness 可以横跨整个轨迹做检测
- 触发检测跟循环检测、上下文压缩是同一层抽象

**④ 具体案例走查 —— 以 packaging 版本语义修复任务为例**：
- 没有 Skill：agent 改了五六次，每次都基于对 API 返回值的错误假设，反复微调反复挂
- 有 Skill：第 3 轮 harness 检测到"同一文件的同一段代码被反复改、同一个测试断言反复失败"→ 自动注入一段提示："你反复修改这段代码但测试持续失败。你可能对这个 API 的返回值理解有误。暂停修改，先用执行环境查一下实际行为"→ agent 查询真实返回值 → 两轮后修复
- 定位：这个 Skill 没有告诉它"该改哪一行"——那是规则版 baseline 的事。Skill 做的是在它陷入低效循环时，把下一步从"继续盲猜"扭成"先去查一下事实"

**⑤ Skill 内容的三个来源（按信任级别分层）**：

| 来源 | 信任级别 | 说明 |
|---|---|---|
| 手工编写 | 最高 | 人把已知的最优调试模式写成 Skill——从成功案例里提炼 |
| 轨迹自动提取 | 中等 | 系统回头分析成功任务的轨迹→提取关键动作序列→抽象成"成功模式"→下次遇到相似特征任务自动加载。就是面经里提到的"自动沉淀"机制 |
| 模型自行总结 | 最低，需审核 | Agent 自己总结自己为什么成功易带认知盲区——"它说策略正确"可能只是碰对的。模型总结的 Skill 进审核队列，人确认没毛病后才激活 |

**面试官简评**：通过。三个关键判断有说服力：(1) "加载时机"选了索引常驻+按需注入的中间方案，没有走全量灌入或纯工具调用的极端；(2) "不应该让模型自己判断什么时候用 Skill"——给出的理由充分（agent 在当前轮次看不到全局轨迹），且跟前面设计的循环检测属于同一层抽象，一致性在；(3) Skill 内容的三来源给了不同信任级别而非笼统地说"都可以"。task_048 的走查展示了 Skill 不是"告诉 agent 答案"而是"在它兜圈时塞一个更优的下一步动作"——这个定位精准。

## Q17: 应用层沙箱隔离 vs Docker 容器隔离 —— 设计哲学与信任模型的根本分歧

**面试官追问背景**：面试官拾起最初自我介绍中"harness 先行 + 四层沙箱"埋下的 hook，要求应聘者做概念层面而非功能清单层面的对比——"你这种文件系统级隔离和 Docker 容器隔离，在设计哲学和信任模型上，根本的分歧在哪里？"

**应聘者回答要点**：

**① 根本分歧：信任模型的方向是反的**：

| 维度 | 当前系统 | Docker |
|---|---|---|
| 信任模型 | **乐观信任**（黑名单）——假设大部分操作安全，只拦已知的危险操作 | **零信任**（白名单）——默认什么都不能做，只放行已知需要的能力 |
| 应用位置 | 应用层——代码检查"不准 import、不准用双下划线" | 内核层——mount/PID/network namespace 里本来就没有那些资源，不需要代码检查 |
| 升级方式 | 被动——发现新绕过方式 → 加一条规则 | 机制性——内核 namespace 实现变强，整个社区在审计 |
| 故障后果 | 沙箱代码出 bug → 一个检查漏了 → agent 出去了 | 内核 namespace 实现有漏洞 → 影响所有用户 → 整个社区驱动修复 |

**② 核心洞察**："每一条黑名单规则都在说'你别做这件事'——但只能拦住我知道要拦的东西。如果有一种攻击方式我没见过、没列在上面，它就过去了。Docker 反过来——一个容器里的进程看不到宿主文件，不是因为有代码在检查，是因为内核 mount namespace 里压根就没有那些文件。"

**③ 为什么知道 Docker 更安全却不用？——决策条件而非技术优劣**：
- 当前场景：单租户评测平台，受信数据源，自己写的测试。威胁模型是"模型随机生成了有问题的代码意外造成破坏"——不是"有人刻意构造攻击"
- 应用层黑名单够拦住"意外"——模型不会刻意绕过 AST 检测去构造逃逸链
- 升级条件：当平台从"单租户"变"多租户"——任意任务提交、任意测试代码执行——威胁模型从"防意外"升级为"防恶意"——这时 Docker/podman + cgroups 是必须的
- 关键判断：**"两个隔离范式不是技术优劣的问题，是信任模型需要覆盖到什么程度的问题。信任越少，隔离层就要越低——从应用层下沉到内核层"**

**④ 当前系统的隔离全图**（honest assessment）：

| 隔离项 | 方式 | 软肋 |
|---|---|---|
| Python 代码执行 | 四层沙箱（文本→AST→空 builtins→截断） | 信任模型是黑名单——乐观信任，防不了未发现的绕路 |
| 文件系统 | `shutil.copytree` 副本 + `ensure_within_boundary()` 路径检查 | 同级目录拷贝，非内核级 namespacing |
| 资源 | `subprocess.timeout` | 只有一个超时——没有内存限制、没有 CPU cgroups |

**面试官简评**：通过。没有停留在"Docker 更安全"的表面比较上，而是把两种隔离的本质提炼成了"信任模型的方向"——乐观 vs 零信任，黑名单 vs 白名单。更重要的是给出了一个清楚的决策条件：什么时候该从应用层下沉到内核层？答案是当威胁模型从"防意外"升级到"防恶意"——也就是从单租户受信环境变成多租户不可信环境的时候。"我清楚地知道这条线画在哪"这句话本身就是一个工程师对自己系统边界有清醒认识的信号。

## Q18: 三重 AND 的实际误判案例 —— 判定边界的两个方向

**面试官追问背景**：面试官重拾最初埋下的 hook——"三重 AND 判定有没有误判过？"不是问理论上会不会，是问 149 次 run 里实际有没有把"修好了"标成失败，或者把"没修好"标成成功。如果没有，敢说只是因为 66 条任务还没撞到边界吗？

**应聘者回答要点**：

**① 方向一：误判成失败（有）——计数器实现 bug**：

- 案例：agent 写了正确修复、测试通过，但跑完测试后额外调了一次读文件确认细节
- 根因：读文件在代码层面被不当归入写操作——generation 计数器被错误加一。验证代际 = 2，workspace 代际 = 3 → 代际不匹配 → 标失败
- 本质：不是判定逻辑设计错了，是实现时读写操作的代际变更边界没对齐
- 如何发现：不是从聚合数字发现的（成功率从 91.8% → 91.3%，不会为一条任务翻 trace），而是应聘者知道那条任务难度不高理论上不该挂，主动追查

**② 方向二：误判成成功（有）——结构性盲区**：

- 案例：agent 误解了 issue，以为 bug 在 A 模块——在 A 改了代码，恰好改变程序执行路径绕开了 B 模块里的真正 bug。测试通过。三重 AND 全满足 → 标 success
- 但几轮后手动 review patch 时发现：这个修复在逻辑上不对——不是在修 bug，是"让测试看不到 bug"。如果跑原仓库完整测试套件，这个 patch 会引入新失败
- 暴露的盲区：**三重 AND 验证的是"测试是否通过了"，不是"修复是否正确"。当测试用例恰好能被一个错误修改满足时，系统无法区分"修好了"和"绕开了"**
- 这个方向没被系统化解决——要解决需要在测试之外加语义验证层，正是前面讨论的"用另一个模型审 patch"的用武之地

**③ 诚实的边界承认**：

三重 AND 的设计假设——测试用例能准确反映 bug 是否被修复——在 66 条 curated 任务里大部分成立，但不是无懈可击的。三重 AND 的防线覆盖情况：

| 方向 | 防线 |
|---|---|
| "agent 什么都没做但系统说做了" | ✅ patch 存在是硬条件——封死了 |
| "agent 做了但没验证就停了" | ✅ 代际匹配封死了 |
| "agent 做了、验证了、但做的是错事" | ⚠️ 只有测试结果这一道防线——测试过了就放行 |

**核心结论**："好的判定系统不是把所有错误都拦住——世界上没有这样的系统。好的判定系统是让你准确地知道'我拦了什么、我没拦什么、剩下那一块没拦住的风险在哪个方向上'。"

**面试官简评**：通过。两种方向的误判各给了一个真实 case，没有只说成功案例。计数器 bug 的发现路径（"这条任务难度不高，理论上不该挂"→ 翻 trace）说明应聘者对每条任务的难度有隐性基线——跑多了磨出来的直觉。"curated 任务"这个表述本身就是对自建评测集天然局限的清醒认识。最后的总结"好的判定系统是让你知道拦了什么、没拦什么、风险在哪个方向"——把安全工程中的"安全声明"思维用到了评测系统上，视角高度够。

## Q19: 无标注、无反馈情况下的抽检策略 —— 从"随机抽"到"按风险排序"

**面试官追问背景**：承接 Q18 "改错地方但测试绕过了"这类误判是靠手动翻 trace 发现的——66 条能翻，660 条不可能条条翻。面试官问：面对一批新 run 结果，在没有用户反馈和人工标注的情况下，怎么快速判断这批结果的质量？有什么抽检策略？

**应聘者回答要点**：

**① 四优先级排序（不是随机抽，是按"最可能出问题的成功"排队）**：

| 优先级 | 筛选条件 | 危险信号 |
|---|---|---|
| P1 | 刚好擦过成功线 | max_iterations=13 却在第 12 轮跑通；patch 只改一行且改动位置跟 bug 描述关键词不沾边 |
| P2 | 收敛轨迹异常 | 改了四五次、每次方向不同、最后才绿——修对了但不确定是不是理解了为什么修对 |
| P3 | 与经验基线偏差 | 简单任务花了 12 轮才完成；或通常需要 10 轮的任务某次 3 轮就完成——"异常短链"也可能是绕过了核心 |
| P4 | 同任务多次 run 结果不一 | 4 次 failed + 1 次 success → 那条 success 更可疑，证明了"这条任务通常过不了" |

**② P2 的具体操作信号**：成功 run 也看写入次数。一次写入就通过的基本不查；三次以上写入才通过的每一例都要看。

**③ 系统打标签 → 人工挑选（可扩展到 660 条）**：

三个自动化标签：
- **边缘度标签**：当次轮数 vs 该任务历史平均轮数——偏离越大排越前
- **稳定性标签**：修复过程的写入次数——越多排越前
- **一致性标签**：多次 run 之间 success/failed 的差异——越不一致排越前

三个标签都是从结构化轨迹里自动算出来的——不需要模型推理，只需要计数和比较。

**④ 核心原则**："抽检的本质不是每条都看，是把最可能藏着问题的那些排在列表最上面——不是抽检，是排序。如果一个系统真的给了我 660 条未标注 run，我花一个下午看完前 30 条就够了——前提是排序是对的。"

**面试官简评**：通过。将"抽检"重新定义为"按风险排序"而非随机抽样——正确。四个优先级的排布有内在逻辑：P1/P2 是过程信号（成功来得多艰难），P3 是历史偏差信号，P4 是一致性信号。三个自动化标签"不需要模型推理只需要计数和比较"——务实地优先做了简单可做的事。搁置点：P3 依赖历史基线，面对全新任务会空转——但 P1/P2 不依赖历史数据，核心排序逻辑仍在。

## Q20: 单 Agent vs Multi-Agent —— 拆分的决策原则 & "领地制"设计

**面试官追问背景**：面试官问如果系统要扩展到跨文件重构任务（需要同时改三个模块还要保证接口兼容），单 agent 还够用吗？如果拆成 multi-agent，依据什么原则决定拆不拆？

**应聘者回答要点**：

**① 拆分原则 —— 不是看任务大小，是看"上下文是否互斥"**：

单 agent 共享上下文在子问题互相冲突时会成为阻碍——模型倾向于用一个思路统一处理所有事，取舍模糊。拆分原则：**当子任务之间"需要同时持有但互不兼容的上下文"时就应该拆**——拆的本质不是"人多力量大"，是给每个 agent 一个不被其他子任务信息污染的局部上下文。

如果子问题是串行依赖的（先改 A → 再让 B 适配 → 再改 C），单 agent 可以处理。如果子问题解耦但要求同时进行、且局部上下文互不兼容，就该拆分。

**② 拆分架构 —— "领地制"而非"主从结构"**：

不拆成"做规划的 + 做执行的"——主从结构等于把 Plan-then-Execute 的弱点放大到多 agent 层（首次规划容易漏关键步骤）。

领地制：每个 agent 负责一个模块或文件，共享全局目标但不共享局部上下文。交互方式——agent 改接口时不直接改别人的代码，而是把改动声明成"接口变更提案"，另一个 agent 拿到提案后在自己的局部上下文里验证、适配。

如果适配冲突，不靠 agent 之间协商——靠 harness 层的 git merge + 测试全量回跑来判断谁的问题。

**③ 核心哲学一致性**："不在 agent 内部解协调问题，把协调问题推到 agent 外部的确定性层——就是 harness + 测试。Agent 管决策，harness 管冲突。"

**④ 对 multi-agent 的态度**：
- 不因为是"热点"就上——当前 66 条任务不需要
- 也不因为"现在够用"就永远不准备拆
- 在等一个信号——第一次出现单 agent 因为上下文污染而系统性失败的时候，拆的理由就充分了

## Q21: "接口变更提案"的结构化设计 —— 无协商 multi-agent 的核心协议

**面试官追问背景**：Q20 中应聘者说 agent 之间的协调不靠协商，靠"接口变更提案"。面试官追问：这个提案到底长什么样？是 agent A 用自然语言描述，还是系统从 patch 自动提取？如果是前者等于又回到了协商问题；如果是后者怎么保证提取不出错？

**应聘者回答要点**：

**① 两个常规方案的问题分析**：

- **自然语言描述（agent A 自己写）**：如果 agent A 说"把 base_version 改成了 public"，agent B 读到这句话可能理解为"把调用改掉"也可能理解为"检查兼容性"。同一条描述在不同上下文里有不同工程含义。不确定性从 agent A 传给了 agent B——协商问题只是换了人承担。

- **从 patch 自动提取签名变化**：patch 告诉系统"从 A 变成 B"，但不告诉系统"哪些变化是故意的、哪些是顺便的"。agent 可能在改接口同时顺手改了变量命名风格。如果整套 patch 签名变化都灌给 agent B，agent B 会过度适应——分不清哪些必须适配、哪些无所谓。

**② 方案：结构化声明工具（agent → 工具 → 系统 → agent B）**：

给 agent 加一个专门的"声明工具"，调用格式是结构化的——不靠自然语言，也不靠事后提取。必须填三件事：
- **改了哪个公开符号**——函数名、类名、方法签名
- **旧行为 vs 新行为**——用输入输出例子，不用自然语言。如"旧：`Version('1.0.dev0').base_version` 返回 `'1.0.dev0'` → 新：返回 `'1.0'`"。不是从 patch 自动提取的，是 agent 改了代码后自己调这个工具填的
- **预计哪些调用方受影响**——agent A 搜索到的受影响文件列表

**③ 关键约束**：agent A 如果不调这个工具，它的跨文件写入操作不被系统接受——不是让 agent 选要不要声明，是 harness 层强制要求。

**④ Agent B 拿到的不是 agent A 说的话**：是系统把上面三条结构化信息直接灌到 agent B 的当前上下文里，格式是系统生成的。agent B 不需要"理解 agent A 在说什么"——它看到的是同一个系统格式的变更通知，就像看测试失败输出一样：这是一个事实，你需要适配。

**⑤ 核心设计意图**："agent A 和 agent B 之间没有直接对话。Agent A 对工具说话，系统把工具输出转换成 agent B 的上下文。协调不是靠 agent 之间沟通，是靠工具产出的结构化事实。agent 管决策，harness 管协调。"

**面试官简评**：通过。把接口变更提案从"agent 间对话"降级为"agent 对工具说话 + 系统传递结构化事实"——这个降级避免了自己前面指出的"协商问题换人承担"的陷阱。强制声明约束（不调工具则写入不被接受）跟三重 AND（没验证不算完）是同一设计范式——不让 agent 选要不要遵守协议，而是系统在协议层强制执行。搁置点：假设 agent A 的声明是可信的——如果 agent A 改错了但不自知，声明本身可能就错了，agent B 基于错误声明做适配。这是整个"无协商"设计的一个未闭合的信任缺口。

## Q22: 声明可信度缺口 —— "无协商"multi-agent 的结构性代价与双层容错

**面试官追问背景**：面试官捡起 Q21 留下的钩子——agent A 的声明本身可能是错的（它改错了但不自知，声明的"旧行为→新行为"描述就是错的），然后 agent B 基于错误声明做适配。应聘者在 Q18 亲口说过"agent 做了、验证了、但做的是错事"——这两件事在 multi-agent 方案里是冲突的。怎么解决？

**应聘者回答要点**：

**① 核心论点：multi-agent 没有引入新问题，只是让旧问题更可见**：

单 agent 里，"以为 bug 在 A 模块"就是一个隐式的内部声明——没人看到它、但它驱动了后续所有行为。Q18 那个"改错地方但绕过了测试"的 case 就是隐式声明错误的后果。Multi-agent 加声明工具只是把这个隐式判断外化了——问题的结构没变，变的是可见性。看起来更糟了（agent B 可能被误导），但实际上是让本就存在的风险更透明。

**② 第一道防线：发送端交叉校验（声明 vs patch）**：

系统拿声明工具的输出跟 agent A 的实际 patch 做对照。检查：声明里提到的函数名在 diff 里出现吗？改的行数跟描述的行为变化一致吗？如果声明说"改了函数 X 的返回值"但 diff 里函数 X 根本没动 → 标"不一致"，不传给 agent B。

交叉校验靠的是文本匹配，不是模型推理。抓得住的：声明跟 patch 在文本层面不一致。抓不住的：声明跟 patch 在文本层面一致（函数 X 确实被改了、返回值确实变了）但语义描述错误（说返回值从 A 变成 B 但实际是从 A 变成 C，且 B/C 碰巧在测试里表现相同）。**缩小了缺口，没封死。**

**③ 第二道防线：接收端行为验证（agent B 的测试）**：

Agent B 不是拿到声明就盲目执行。它自己的局部上下文里也要跑测试。如果 agent A 的声明错了，agent B 基于错误声明做的适配大概率在 B 的测试里暴露。即使错误声明穿过了交叉校验，B 的测试会挂。挂了之后的 trace 完整记录了"agent A 声明了什么 → agent B 据此做了什么 → 测试挂了什么"——错误链可追溯。

**④ 结论**："我接受'声明可能出错'是这个体系的结构性代价——就像我接受单 agent 的'修复可能是错的'一样。但剩下那个消不掉的缺口，就是'agent 的语义理解错误'——这是所有 agent 系统的共同天花板，不是 multi-agent 独有的。"

**⑤ 双层容错架构总结**：

| 防线 | 位置 | 能力 | 盲区 |
|---|---|---|---|
| 交叉校验 | 发送端（A → 系统） | 文本层面声明与 patch 的一致性 | 语义层面——声明和 patch 文本一致但语义错误 |
| 行为验证 | 接收端（B 自己的测试） | 基于声明做出的适配是否正确 | B 的测试覆盖不全时可能漏过 |

两个加在一起，把信任缺口从"可能随便错"缩到"需要同时骗过交叉校验和 B 的测试"。

**面试官简评**：通过。核心贡献是把 multi-agent 的"声明错误传播"问题重新框定为"单 agent 里本就存在的隐式错误的外化"——不是否认问题，是指出问题不是新引入的。交叉校验 + B 的测试构成两道防线——防御纵深而非单点信任。坦率画了"抓得住 vs 抓不住"的分界线，没有过度承诺。"所有 agent 系统的共同天花板"这个结论将具体问题升维到了认知边界层面。
