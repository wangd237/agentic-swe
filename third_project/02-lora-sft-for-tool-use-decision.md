# 第三个项目方向方案二：LoRA SFT for Tool-Use Decision

## 方向定位

这个方向适合作为前两个项目的“训练增强版延伸”。

如果前两个项目已经完成了：

1. `Mini SWE-Agent for GitHub Issues`
2. `Tool-Use and Patch-Quality Optimization for SWE-Agent`

那么这个第三项目可以继续往前走一步：

- 不只是靠 prompt 和规则来优化 Agent
- 而是尝试用小规模训练数据，直接改进模型在工具使用和修复决策上的行为

它会成为你整条主线里的“轻量后训练实验”部分。

---

## 这个项目在做什么

简单来说，这个方向是在做：

`让模型更会做 SWE-Agent 里的关键决策`

这里的“关键决策”指的不是让模型懂所有编程知识，而是让它在一些具体行为上更稳定，例如：

- 先找测试还是先找实现文件
- 搜什么关键词更合理
- 什么时候应该停止搜索
- 什么时候应该开始改代码
- 改完后是否会主动跑测试
- 面对不确定情况时是否更谨慎

这个项目不是去训练一个全新的大模型，而是通过 `LoRA + SFT` 这种相对轻量的方式，让模型在 `tool-use decision` 这类窄任务上表现更好一点。

---

## 它和前两个项目怎么结合

这个结合关系也非常顺：

- `项目 1` 提供 Agent 的运行环境和真实执行任务
- `项目 2` 提供评测指标和失败类型分析
- `项目 3` 则把这些失败样本转化成训练数据，做轻量行为优化

整个闭环会变成：

`Agent 执行 -> 轨迹收集 -> 错误分析 -> 构造训练数据 -> LoRA SFT -> 再评测`

这条线非常像真实大模型团队里的 post-training mini 版本。

---

## 这个方向主要关注什么

它不是重点优化“知识量”，而是重点优化“行为模式”。

也就是说，你更关注的是：

- 是否更会选工具
- 是否更会安排修复顺序
- 是否更会减少无效步骤
- 是否更会做出小而准的 patch

它关注的是 Agent 的决策质量，而不是纯问答准确率。

---

## 为什么这个方向值得放简历

因为它能直接补上“训练能力”这块标签。

你的前两个项目已经足够证明：

- 你会搭 Agent
- 你会做评测
- 你会做优化

但如果再加上这个方向，你就还能说：

- 我会从 Agent 轨迹中构建数据
- 我会做 LoRA / SFT
- 我会比较训练前后的行为变化

哪怕规模不大，这个信号对很多实习岗位也很有价值，因为它说明你不仅会做应用，还接触过 `post-training` 的基本工作流。

---

## 适合怎么呈现

这个项目适合被定义成：

`面向 SWE-Agent 工具使用决策的小规模后训练实验`

它不是一个完全独立的新系统，而是前两个项目的自然加深：

- 前两个项目负责让 Agent 跑起来并评估它
- 这个项目负责把评估得到的问题进一步转成训练改进

最终可以讲成：

> 基于 SWE-Agent trajectory 构造监督数据，围绕工具选择、测试意识和 patch 行为开展 LoRA SFT 实验，探索轻量 post-training 对 agent 决策质量的影响。

---

## 它最补的能力是什么

它最补的是：

- `training signal`
- `SFT / LoRA`
- `data construction`
- `post-training thinking`

如果你想让简历更像“agent + 训练”路径，这个方向会非常契合。

---

## 它的整体风格

这个项目的风格不是“重新做一个新应用”，而是：

`在已有 Agent 系统之上，增加一个轻量训练与行为优化层。`

所以它和前两个项目的衔接会非常强，也最容易讲成一条完整主线。
