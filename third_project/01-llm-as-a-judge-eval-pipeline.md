# 第三个项目方向方案一：LLM-as-a-Judge Eval Pipeline

## 方向定位

这个方向适合作为前两个项目的自然延伸。

如果前两个项目分别是：

1. `Mini SWE-Agent for GitHub Issues`
2. `Tool-Use and Patch-Quality Optimization for SWE-Agent`

那么第三个项目 `LLM-as-a-Judge Eval Pipeline` 做的事情就是：

- 给前两个项目补上一套更智能、更系统的自动评测层
- 让你的整个主线从“能做 Agent、能优化 Agent”进一步升级为“还能自动评价 Agent”

它会成为整个大项目里的“判卷系统”。

---

## 这个项目在做什么

简单来说，它是在做一个“AI 评委”。

前两个项目已经可以让 Agent 去读仓库、改代码、跑测试、输出 patch。  
但很多质量问题，不能只靠“测试过没过”来判断，例如：

- 它虽然修好了，但 patch 改得太大了
- 它虽然跑了测试，但步骤很绕，读了很多无关文件
- 它虽然看起来给出了答案，但实际上定位思路不合理
- 它虽然没有完全修好，但分析路径是对的

这些问题，正是 `LLM-as-a-Judge` 要解决的。

它会尝试自动回答：

- 这个 Agent 的轨迹是否合理？
- 这个 patch 是否聚焦问题？
- 这个修复过程是否高效？
- 这个最终结果是否值得判为“高质量修复”？

---

## 它和前两个项目怎么结合

结合方式非常自然：

- `项目 1` 负责产生任务执行结果和轨迹
- `项目 2` 负责定义一部分手工指标和优化方向
- `项目 3` 负责用 LLM Judge 对这些结果做更细粒度的自动评价

于是整个主线会变成：

`SWE-Agent 执行 -> 轨迹记录 -> 指标统计 -> LLM Judge 评分 -> 错误分析 -> 优化`

这样你的项目结构就比普通 Agent 项目完整得多。

---

## 这个方向主要关注什么

这个项目不直接修 bug，而是评估“修 bug 的过程和结果”。

重点关注的维度大概包括：

- 轨迹是否合理
- 工具使用是否有明显冗余
- 是否读到了关键文件
- patch 是否集中在真正相关位置
- 是否进行了必要测试
- 最终回答是否诚实、完整、可解释

它会比单纯的 `success rate` 更细，也更接近研究和平台团队的工作方式。

---

## 为什么这个方向值得放简历

因为它能体现出你不仅会“做系统”，还会“评系统”。

很多候选人的项目停留在：

- 做了一个 Agent
- 看起来能跑
- 展示几个 case

而这个方向能把你的项目提升到：

- 我能定义评测维度
- 我能设计 Judge
- 我能分析 Judge 与人工判断的一致性
- 我能把 Judge 用到大规模实验和优化闭环里

这会让你的项目更像真正的大模型实验项目，而不是纯 demo。

---

## 适合怎么呈现

这个项目适合被写成前两个项目的“评测增强层”。

它不是孤立存在的，而是：

- 输入来自 SWE-Agent 的运行日志、patch、测试结果
- 输出是结构化评分、错误标签、质量报告

最终可以讲成：

> 在 Mini SWE-Agent 基础上，构建基于 LLM 的自动评测管线，用于对 agent trajectory、tool use 和 patch quality 进行细粒度评分，并辅助后续优化实验。

---

## 它最补的能力是什么

它最补的是：

- `evals`
- `metrics thinking`
- `experiment design`
- `LLM-as-a-judge`

如果你想让简历更偏大模型评测、Agent 平台、模型质量分析，这个方向很合适。

---

## 它的整体风格

这个项目的整体风格不是“再做一个应用”，而是：

`在已有 Agent 项目之上，增加一个高价值的评测研究层。`

所以它和前两个项目是强关联、强互补的。
